# Frontend Design Doc Generator

根据 PRD、原型/UI 设计、现有代码库和开发者补充信息，生成可直接指导前端开发的项目设计书。

## 何时使用

- 用户提供 PRD、CoDesign/蓝湖/Figma/截图等原型或 UI 信息，需要输出前端设计书
- 用户要求把需求拆成前端模块、页面、路由、组件、接口和状态流转
- 用户已有设计书初稿，需要按统一结构补全或重写
- 用户要求对照现有项目技术栈、组件库、路由和 Store 模式生成开发方案

## 输入材料

优先收集以下信息，不完整时也可以先产出草稿，并把缺失项标为 `TODO`：

| 材料 | 用途 | 缺失时处理 |
| --- | --- | --- |
| PRD 需求文档 | 业务逻辑、字段定义、权限矩阵、生命周期规则、审批规则 | 标注 `TODO：待 PRD 确认` |
| 原型/UI 设计 | 页面布局、交互细节、组件样式、状态流转 | 标注 `TODO：待原型确认` |
| 现有代码库 | 技术栈、组件库、路由约定、Store 模式、接口封装方式 | 读取代码后对齐现有实现 |
| 模块划分 | 章节顺序、优先级、开发边界 | 先基于 PRD 自动拆分，再请用户确认 |
| 开发者补充 | 特殊约束、字段命名、接口约定、兼容逻辑 | 作为最高优先级输入 |

如果链接需要登录（如语雀、CoDesign、Confluence、私有 GitHub），不要使用 WebFetch 直接抓取；请让用户提供可访问内容、截图、导出的 Markdown/PDF，或在有浏览器/MCP 授权时通过对应工具读取。

### MasterGo MCP 读取规则

当输入材料包含 MasterGo 设计稿链接时，优先使用 `mastergo-magic-mcp`，不要退化为截图猜测。读取顺序如下：

1. 先解析链接参数：`fileId` 取 `/file/<fileId>`；目标图层只接受 `layer_id` 或 `source_layer_id`；`page_id` 只表示页面，不等同于图层 ID。
2. 如果链接是短链或包含 `layer_id`，优先调用 `mcp__mastergo_magic_mcp__.mcp__getDsl` 获取 DSL；需要页面级站点信息时，再调用 `mcp__getMeta`。
3. 如果 `getDsl` 返回 `componentDocumentLinks` 且非空，必须逐个调用 `mcp__getComponentLink` 获取组件文档，再结合 DSL 生成设计书。
4. 如果链接只有 `page_id`，不得把 `page_id` 当作 `layerId` 下结论；应标记为 `TODO：待提供带 layer_id 的 MasterGo 图层链接`，或在有浏览器授权时通过 Chrome/Computer Use 辅助用户复制“选中图层链接”。
5. 如果 MasterGo MCP 返回空 `nodes/components/styles`，按证据记录为空结果，不能据此臆造页面结构；需要请求具体画板、容器或组件的 `layer_id`。
6. 生成设计书时，视觉信息必须来自 DSL、组件文档、PRD 或用户补充；缺失字段、接口、权限码统一标 `TODO` 或 `暂定`。

## 工作流程

### 步骤 1：确认输入范围

1. 识别用户给出的 PRD、原型、UI、旧设计书、模块划分。
2. 如果用户只给链接且无法访问，说明限制并请求粘贴关键内容或提供本地文件。
3. 如果已有设计书初版，先复用其模块和业务信息，再按本 skill 的结构重排补齐。
4. 不要臆造不存在的接口路径、字段 key、权限码；不确定内容统一标 `暂定` 或 `TODO`。
5. 如输入包含 MasterGo 链接，先按“MasterGo MCP 读取规则”完成 DSL/Meta/组件文档读取；无法取得 `layer_id` 时先记录 TODO，不进入细节推断。

### 步骤 2：读取现有代码库约定

如用户要求“贴合项目代码”或当前目录是代码库，优先检查：

- 技术栈：Vue/React、Vite/Webpack、TypeScript、状态管理
- UI 组件库：表单、表格、弹窗、抽屉、上传、字典组件
- 路由约定：列表页、详情页、新建/编辑/审批是否复用路由
- 列表页封装：如 `createPage`、分页字段、查询按钮、表格配置方式
- 权限控制：如 `useAuth`、`hasAuth`、权限码组织方式
- 请求封装：接口目录、API 方法命名、错误处理方式
- Store 模式：Pinia/Vuex/Zustand/Redux、是否持久化

输出时只能引用已确认的现有约定；未确认时写成“建议”或 `TODO：待代码库确认`。

### 步骤 3：模块拆分

按业务闭环拆模块，而不是按接口或组件拆模块。每个模块至少明确：

| 维度 | 要回答的问题 |
| --- | --- |
| 页面 | 有哪些列表、详情、新建、编辑、审批、弹窗、抽屉 |
| 数据 | 页面初始化、查询、提交、审批、导出、上传的链路 |
| 状态 | 草稿、申请中、已通过、已驳回、已撤销、待执行等状态如何影响 UI |
| 权限 | 按钮级、行级、数据级分别怎么控制 |
| 复用 | 复用哪些现有组件，哪些需要新建公共组件 |
| 风险 | 哪些字段、接口、规则需要后端/产品确认 |

### 步骤 4：按固定章节输出

每个业务模块必须包含以下 5 个子章节，格式参考 [输出模板](./references/output-template.md)：

1. 页面结构
2. 数据链路
3. 组件复用/沿用
4. 相关接口
5. 补充逻辑

公共组件、公共权限、公共路由规范、公共数据字典放在“公共约定”章节，模块内只引用，不重复长篇定义。

## 输出要求

### 文档结构

推荐顶层结构：

1. 基础信息
2. 版本履历
3. 需求概述
4. 前端模块总览
5. 公共约定
6. 业务模块设计（逐模块）
7. 路由汇总
8. 接口清单
9. TODO / 待确认事项
10. 测试要点

### 每个模块必须覆盖

#### 1. 页面结构

必须包含：

- 路由定义：路径、query 参数、同一路由复用情况
- 页面层级关系：列表 → 详情 → 弹窗/抽屉/审批区，用树形结构表示
- 布局逻辑：用 ASCII 图表示筛选区、操作栏、表格区、分页栏、详情区
- 元素排布：筛选项、操作按钮、表格列、底部操作区的位置关系
- 交互细节：悬停、点击、拖拽、键盘操作、右键菜单、二次确认、未保存拦截

#### 2. 数据链路

必须包含：

- 页面初始化加载哪些接口
- 查询/筛选/分页/排序的请求时机和参数组装
- 新建/编辑/审批/撤销/导入/导出的请求链路
- Store 缓存字段、是否持久化、何时清空
- 数据变更后的联动刷新逻辑

#### 3. 组件复用/沿用

必须包含：

- 复用现有组件，并标注来源：组件库、业务公共组件、当前模块组件
- 需要新建的组件：中文组件名、职责、props、emit、使用场景
- 组件间通信方式：props、emit、provide/inject、Store、路由 query

组件名称使用中文，如 `<商品选择器 />`，不要只写英文组件名。

#### 4. 相关接口

必须包含接口表格：

| 接口名称 | 方法 | 路径 | 入参 | 出参 | 调用时机 | 备注 |
| --- | --- | --- | --- | --- | --- | --- |

如果接口未定，路径写 `TODO：待后端确认`；字段 key 未定，标注 `暂定`。

#### 5. 补充逻辑

必须包含：

- 边界条件：空数据、超长文本、重复提交、并发操作、接口失败
- 权限控制：按钮级、行级、数据级
- 校验规则：前端表单校验、业务规则校验、跨字段校验
- 性能考量：虚拟滚动、防抖节流、懒加载、批量操作分片
- 审批/生命周期/状态机等额外业务规则

## 编写规范

- 表格优先于长文本，列表优先于大段叙述。
- 字段、接口、权限码不确定时不要编造，统一标 `暂定` 或 `TODO`。
- 路由详情页与编辑页优先复用同一路由，通过 `?id=xxx&mode=edit` 或 `?mode=view` 区分，除非项目已有不同约定。
- 列表页优先沿用项目现有列表构建方式；如果项目已有 `createPage`，明确写“使用 createPage 构建”。
- 权限控制优先沿用项目现有方式；如果项目已有 `useAuth` / `hasAuth`，在按钮级、行级、数据级分别说明。
- 公共组件集中放在“公共约定”，模块章节只引用。
- 原型中的视觉信息要落到前端可执行描述：组件、布局、交互、状态、校验，不只描述“按原型”。
- 对 PRD 与原型冲突的内容，单独列“冲突/待确认事项”。

## 质量检查清单

输出前检查：

- 每个模块是否都有 5 个固定子章节
- 每个列表页是否有筛选区、操作区、表格列、分页/排序说明
- 每个详情/编辑/新建页是否有路由、表单字段、校验、保存/提交链路
- 每个审批页是否有状态、审批人、操作按钮、通过/驳回链路
- 每个导入/导出/上传是否有文件格式、触发时机、错误反馈
- 是否列出接口、字段、权限、原型缺失的 TODO
- 是否对齐现有技术栈和组件库，而不是输出通用方案
- 如果使用 MasterGo，是否记录 `fileId`、`layer_id/source_layer_id`、MCP 返回状态，并避免把 `page_id` 当作图层 ID

## 输出风格

- 使用中文输出。
- 保持可直接复制进设计书的 Markdown 格式。
- 对长文档先输出完整目录和一个模块样例；用户确认后再继续补齐剩余模块。
- 如果用户明确要求一次性完整输出，则完整生成，不省略章节。
