# Frontend Design Doc Generator v4.0.0

根据 PRD、原型/UI 设计、现有代码库和开发者补充信息，生成 **AI 可独立实现的前端实施设计书**。

## 版本标识（必须遵守）

本 skill 当前版本为 **v4.0.0**。生成的每份设计书**必须**在基础信息表格中包含以下追溯字段：

| 追溯字段 | 值 | 说明 |
| --- | --- | --- |
| 生成工具 | frontend-design-doc-generator | 固定值 |
| 工具版本 | v4.0.0 | 取本 skill 的 version |
| 生成时间 | 当前日期 | 设计书首次生成时间 |

如果输出中缺少这三个字段，说明**没有使用本版本的 skill**。

## 设计目标

本 skill 的输出不是普通前端设计书，而是**编码实施手册**。生成的文档必须让未参与需求讨论的编码 AI，仅凭本文档和当前代码库，即可知道：

| 维度 | 必须能回答的问题 |
| --- | --- |
| 文件落点 | 改哪些文件、新建哪些文件、删除哪些文件 |
| 复用策略 | 复用哪些现有组件/API/Store/工具函数，import path 是什么 |
| 页面实现 | 页面、组件、接口、状态、权限、校验如何落到代码 |
| 字段契约 | 字段 key、类型、默认值、枚举、展示规则、API 映射 |
| 阻塞管理 | 哪些 TODO 阻塞实现，哪些可以暂定或 mock，mock contract 是什么 |
| 实现顺序 | 先做什么、后做什么、依赖关系是什么 |
| 验收标准 | 每个场景的 Given/When/Then 是什么 |

## 何时使用

- 用户提供 PRD、CoDesign/蓝湖/Figma/MasterGo/截图等原型或 UI 信息，需要输出前端设计书
- 用户要求把需求拆成前端模块、页面、路由、组件、接口和状态流转
- 用户已有设计书初稿，需要按统一结构补全或重写
- 用户要求对照现有项目技术栈、组件库、路由和 Store 模式生成开发方案

## ⛔ 结构化输出硬约束（不可违反）

> 本节为 v4.0.0 核心约束。以下规则优先级最高，任何其他指令不得覆盖。

### 量化底线

| 指标 | 最低值 | 检查方式 |
| --- | --- | --- |
| 每模块代码块总数 | ≥ 6 | 路由代码 + API Map + TypeScript interface×2 + Hooks + 组件契约 |
| 每模块 TypeScript interface | ≥ 2 | 请求参数类型 + 响应数据类型 |
| 每模块状态机表格 | ≥ 1 | 有状态流转时必须完整列出 |
| 每模块 ASCII 线框图 | ≥ 1 | 列表/详情/编辑页各一张 |
| 每模块实施蓝图 | = 1 | 每个模块必须包含文件变更清单、依赖顺序、阻塞项 |
| 每模块验收标准 | ≥ 3 | 至少 3 条 Given/When/Then 场景 |
| 代码库证据清单 | ≥ 1 | 步骤 2 必须输出结构化证据清单 |
| 全文 Evidence 引用（代码） | ≥ 15 | 指向真实代码文件的 `Evidence(path:line)` |
| 全文 Evidence 引用（PRD） | ≥ 8 | 引用 PRD 章节/行号 |
| 全文 TODO 总数 | ≤ 15 | 超出说明代码库探索不充分 |
| 阻塞型 TODO | 100% 有临时方案 | P0 TODO 必须含 mock contract 或建议默认方案 |
| 自检清单 | 20 项全列 | 末尾必须追加完整自检表 |

### 禁止项（违反即不合格）

| # | 禁止内容 | 替代方案 |
| --- | --- | --- |
| 1 | 禁止出现"按原型"、"按 PRD"、"参考设计稿"等不落地描述 | 必须转化为：组件、布局、交互、状态、校验的具体描述 |
| 2 | 禁止接口路径写 `TODO` | 必须基于项目现有路由/API 命名约定推导，可标 `(待确认)` |
| 3 | 禁止字段字典用列表或描述性段落 | 必须用 10 列表格 |
| 4 | 禁止省略任何 13 子章节之一 | 无内容时标注"本模块不涉及此项"并说明原因 |
| 5 | 禁止不写代码示例 | 每模块至少 6 类代码块（路由/API Map/Interface×2/Hooks/组件契约） |
| 6 | 禁止编造不存在的文件路径、组件名、权限码 | 不确定的统一标 `(待确认)` |
| 7 | 禁止状态机用纯文字描述 | 必须用表格：状态 / UI 表现 / 允许动作 / 禁止动作 |
| 8 | 禁止遗漏 PRD 中的需求模块 | 必须逐一核对 PRD 需求列表，确保全覆盖 |
| 9 | 禁止 TODO 替代整个章节 | TODO 只能用于具体字段/路径/码值，不能替代章节内容 |
| 10 | 禁止验收标准只写"功能正常" | 必须用 Given/When/Then 格式描述具体场景 |

## 输入材料

优先收集以下信息，不完整时也可以先产出草稿，并把缺失项标为 `(待确认)`：

| 材料 | 用途 | 缺失时处理 |
| --- | --- | --- |
| PRD 需求文档 | 业务逻辑、字段定义、权限矩阵、生命周期规则、审批规则 | 标注 `(待确认：PRD 未提供)` |
| 原型/UI 设计 | 页面布局、交互细节、组件样式、状态流转 | 标注 `(待确认：原型未提供)` |
| 现有代码库 | 技术栈、组件库、路由约定、Store 模式、接口封装方式 | 读取代码后对齐现有实现 |
| API 文档/Mock/OpenAPI | 接口 method/path/request/response/error | 标注 `(待确认：API 文档未提供)`，基于命名约定推导 |
| 设计系统/组件库文档 | 设计 token、组件行为、表单/表格规范 | 读取代码中已有用法推导 |
| 模块划分 | 章节顺序、优先级、开发边界 | 先基于 PRD 自动拆分，再请用户确认 |
| 开发者补充 | 特殊约束、字段命名、接口约定、兼容逻辑 | 作为最高优先级输入 |

如果链接需要登录（如语雀、CoDesign、Confluence、私有 GitHub），不要使用 WebFetch 直接抓取；请让用户提供可访问内容、截图、导出的 Markdown/PDF，或在有浏览器/MCP 授权时通过对应工具读取。

### MasterGo 原型处理

如果用户提供 MasterGo 链接（含 `fileId`、`layerId`），**必须**使用 `mastergo-magic-mcp` 的 `getDsl` 工具读取设计稿 DSL 数据，从中提取：

- 页面宽度、布局方式（flex/grid/absolute）
- 组件层级关系
- 字体、颜色、间距等设计 token
- 交互状态（hover/active/disabled）

将 DSL 信息映射到前端组件和布局描述中，而不是仅写"按原型"。

## 工作流程

### 步骤 1：确认输入范围与访问能力

1. 识别用户给出的 PRD、原型、UI、旧设计书、模块划分。
2. 如果用户只给链接且无法访问，说明限制并请求粘贴关键内容或提供本地文件。
3. 如果已有设计书初版，先复用其模块和业务信息，再按本 skill 的结构重排补齐。
4. **必须**逐一核对 PRD 需求列表，确保所有需求模块都被覆盖（包括迭代优化清单）。
5. 不要编造不存在的文件路径、组件名、权限码；不确定内容统一标 `(待确认)`。

### 步骤 2：读取证据并提取事实

**当项目目录是代码仓库时，本步骤为强制执行。**

必须使用 Agent（Explore 类型）或文件读取工具，扫描以下内容并记录发现。

**强制输出：代码库证据清单**

每个探索项**必须**在设计书中以结构化表格输出：

| 探索项 | 文件路径 | 发现结论 | 复用方式 | import/注册位置 | 可信度 |
| --- | --- | --- | --- | --- | --- |
| 技术栈 | `package.json:xx` | Vue 3 + Pinia + Ant Design Vue | — | — | ✅已确认 |
| 列表页模式 | `pages/xxx/index.tsx:216` | 使用 createPage 构建 | 直接复用 createPage | `import { createPage } from '...'` | ✅已确认 |
| 权限模式 | `pages/xxx/index.tsx:11-19` | codes + hasAuth | 沿用现有模式 | `import { hasAuth } from '...'` | ✅已确认 |
| ... | ... | ... | ... | ... | ... |

**必须探索的项目**：

| 探索项 | 必须读取的内容 | 产出 |
| --- | --- | --- |
| 技术栈 | `package.json`、`tsconfig.json`、构建配置 | 框架版本、UI 库、表格库、状态管理 |
| 目录结构 | `src/` 或 `pages/` 目录树 | 页面组织方式、公共组件位置 |
| 列表页模式 | 找到至少 2 个现有列表页的实现文件 | `createPage` / 自定义列表模式、分页字段 |
| 权限模式 | `hasAuth` / `useAuth` / `codes` 定义方式 | 权限码格式、按钮控制方式 |
| 请求封装 | `request/` 或 `api/` 目录 | APIMap 格式、接口注册方式 |
| 表单模式 | `useFormItem` / 表单项配置方式 | 控件类型、字典消费方式 |
| 相关模块 | 当前需求最相关的页面文件 | 挂载点、已有组件可复用项 |
| 上传/导入 | 现有上传组件、Excel 处理工具 | 可复用的上传/导入能力 |
| 路由配置 | 路由注册文件 | 路由定义格式、懒加载方式 |
| 类型定义 | 现有 typing.ts / types.ts | 类型命名约定 |
| Store/Composable | stores/ 或 composables/ 目录 | Store 定义方式、命名约定 |
| 样式方案 | CSS/Less/UnoCSS 配置文件 | 样式文件组织方式、设计 token |

每个探索项**必须**在设计书中以 `Evidence(path:line)` 格式引用代码位置：

```
Evidence(`apps/pim/pages/preselected/index.tsx:216`)
```

**Evidence 最低要求**：代码引用 ≥ 15 处 + PRD 引用 ≥ 8 处 = 总计 ≥ 23 处。

### 步骤 3：模块拆分与实现范围确认

采用**混合拆分模式**：

```
输出目录/
├── 00_总览与公共约定.md          ← 职能分层（架构+路由+权限+公共组件+技术栈+证据清单）
├── 01_{模块A}_设计书.md          ← 业务模块平铺（13 子章节 + 实施蓝图 + 强制代码块）
├── 02_{模块B}_设计书.md          ← 每个 PRD 功能模块一份独立文件
├── ...
├── N-1_系统集成与兼容.md         ← 横切关注点（新老系统兼容/消息推送等）
└── N_自检与测试策略.md           ← 质量保障（自检20项+验收标准+错误处理）
```

**拆分原则**：
- 总览文件（00）：放所有跨模块公共内容 + 代码库证据清单
- 业务模块文件（01~N-2）：按 PRD 功能域拆分，每模块含完整 13 子章节 + 实施蓝图
- 横切文件（N-1）：系统集成、数据同步、消息通知等跨模块能力
- 质量保障文件（N）：自检清单 + 验收标准 + 错误处理规范

每个模块至少明确：

| 维度 | 要回答的问题 |
| --- | --- |
| 页面 | 有哪些列表、详情、新建、编辑、审批、弹窗、抽屉 |
| 数据 | 页面初始化、查询、提交、审批、导出、上传的链路 |
| 状态 | 草稿、申请中、已通过、已驳回、已撤销、待执行等状态如何影响 UI |
| 权限 | 按钮级、行级、数据级分别怎么控制 |
| 复用 | 复用哪些现有组件，哪些需要新建公共组件 |
| 风险 | 哪些字段、接口、规则需要后端/产品确认 |
| 文件 | 新建/修改/删除哪些文件，文件之间的依赖关系 |

### 步骤 4：生成实施蓝图（每个模块强制执行）

在每个模块的详细设计**之前**，必须先生成实施蓝图：

#### 4.1 文件变更清单

| 变更类型 | 文件路径 | 实现内容 | 依赖 | 参考实现 |
| --- | --- | --- | --- | --- |
| 页面 | 修改 | `apps/pim/pages/xxx/index.tsx` — 新增导入按钮 | 无 | `apps/pim/pages/preselected/index.tsx:289` |
| 组件 | 新建 | `apps/pim/pages/xxx/import/ImportDrawer.tsx` | 类型文件 | `apps/pim/pages/newGoods/Drawer.tsx` |
| API | 修改 | `apps/pim/request/record/goods/xxx/index.ts` — 新增 4 个接口 | 类型文件 | `apps/pim/request/record/goods/preselected/index.ts` |
| 类型 | 新建 | `apps/pim/pages/xxx/import/importTypes.ts` | 无 | `apps/pim/pages/preselected/skuSections.tsx` |
| Store/Hook | 新建 | `apps/pim/pages/xxx/import/useImportLogic.ts` | API + 类型 | — |
| 样式 | 新建 | `apps/pim/pages/xxx/import/ImportDrawer.less` | 无 | — |

#### 4.2 依赖顺序

```text
① 类型定义（无依赖）
  → ② API 层（依赖类型）
    → ③ Store/Hooks（依赖 API + 类型）
      → ④ 子组件（依赖 Hooks + 类型）
        → ⑤ 页面容器（依赖子组件）
          → ⑥ 路由/权限注册（依赖页面）
            → ⑦ 测试验证（依赖所有）
```

#### 4.3 复用映射

| 需求中的页面/组件 | 代码库中对应的现有模式 | 复用方式 |
| --- | --- | --- |
| 导入记录列表 | `createPage` 列表模式 | 直接复用 createPage |
| 文件上传 | `readFiles` + `usePluginImportFiles` | 参考实现，新建专用组件 |
| 权限控制 | `codes` + `hasAuth` | 沿用现有模式 |

#### 4.4 阻塞项

| ID | 阻塞内容 | 是否阻塞实现 | 影响步骤 | 临时方案 |
| --- | --- | --- | --- | --- |
| B1 | 导入记录 API 路径未确认 | 不阻塞 | 步骤 ② API 层 | 基于命名约定推导 `/goods/preselected/import/page`，标 `(待确认)` |
| B2 | 权限码未确认 | 不阻塞 | 步骤 ⑥ 权限注册 | 暂定 `pim:preselected:import`，标 `(待确认)` |

### 步骤 5：按实施级章节输出

每个业务模块**必须**包含以下 **13 个子章节**，格式参考 [输出模板](./references/output-template.md)：

1. **模块证据与范围** — PRD 证据 + MasterGo 证据 + 代码库证据 + 模块边界
2. **文件变更计划** — 完整的实施蓝图（步骤 4 产出）
3. **页面结构与路由** — 路由代码 + 层级树 + 布局 ASCII 图 + 交互细节
4. **UI 布局与交互契约** — 布局规则 + 交互场景 + 反馈 + 空态/异常态
5. **数据模型与字段字典** — 完整 10 列表格 + 自动计算字段 + 联动规则
6. **API 契约** — 12 列完整表格 + API Map 代码块 + TypeScript Interface + API 字段映射
7. **组件复用与新建组件契约** — 复用组件表 + 新建组件契约表（含 props/emits/slots/exposed）
8. **状态、缓存与数据链路** — 状态与缓存设计表 + 初始化/查询/提交/联动链路 + 状态转换图
9. **权限、校验与状态机** — 三层权限表 + 校验规则表 + 状态机表 + 行级状态矩阵
10. **异常、边界与性能** — 异常处理表 + 边界条件表 + 性能策略表
11. **Hooks 与工具函数** — 自定义 Hooks 代码块 + 工具函数代码块 + 使用方式
12. **实现步骤** — 步骤表（操作 + 涉及文件 + 前置依赖 + 完成标准）
13. **验收标准与 TODO** — Given/When/Then 验收表 + 结构化 TODO 清单

**每个模块必须包含的代码块**：

| 代码块类型 | 最低数量 | 说明 |
| --- | --- | --- |
| 路由配置代码 | 1 段 | 完整 `import()` 懒加载路由定义，含路由注册位置 |
| API Map 定义 | 1 段 | 遵循项目现有 `[method, path, ReqType, ResType]` 四元组格式 |
| TypeScript Interface | ≥ 2 段 | 请求参数 + 响应数据的完整类型定义 |
| 自定义 Hooks | ≥ 1 段 | 业务逻辑封装（价格计算/权限判断/状态管理/数据脱敏等） |

## TODO 管理规则

### 结构化 TODO 格式

所有 TODO **必须**使用以下 9 列格式：

| ID | 类型 | 问题 | 是否阻塞实现 | 影响范围 | 临时方案 | 负责人 | 优先级 | 确认后需修改章节 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| T1 | 接口 | 导入记录 API 路径未确认 | 否 | 模块06 API 契约 | 基于命名约定推导 `/goods/preselected/import/page`，标 `(待确认)` | 后端 | P1 | 6. API 契约 |
| T2 | 权限 | 批量导入权限码未确认 | 否 | 模块09 权限控制 | 暂定 `pim:preselected:import`，标 `(待确认)` | 产品 | P1 | 9. 权限控制 |

### TODO 分级规则

| 阻塞实现 | 处理方式 |
| --- | --- |
| **是（P0 阻塞）** | 必须在临时方案中写清 mock contract 和后续替换点；必须标注影响哪个实现步骤 |
| **否（P1 待定）** | 可基于命名约定推导暂定值，标 `(待确认)`；标注确认后需要修改哪些章节 |
| **否（P2 优化）** | 记录即可，不影响主流程实现 |

### TODO 约束

- `TODO` 不能替代整个章节内容。
- P0 阻塞型 TODO 必须明确阻塞哪个实现步骤。
- `暂定` 只能用于结构清晰但名称/字段/路径未最终确认的内容。
- 如果可以先 mock，必须写清 mock contract 和后续替换点。

## 编写规范

- **表格优先于长文本**，列表优先于大段叙述。
- **Evidence 引用**：引用代码库事实时**必须**带 `Evidence(path:line)` 格式；引用 PRD 事实时标注 `PRD:行号` 或章节名。
- 字段、接口、权限码不确定时不要编造，统一标 `(待确认)`。
- 路由详情页与编辑页**优先**复用同一路由，通过 `?id=xxx&mode=edit` 或 `?mode=view` 区分，除非项目已有不同约定。
- 列表页**必须**沿用项目现有列表构建方式；如果项目已有 `createPage`，明确写"使用 createPage 构建"并附 Evidence。
- 权限控制**必须**沿用项目现有方式；如果项目已有 `useAuth` / `hasAuth`，在按钮级、行级、数据级分别说明。
- 公共组件集中放在"00_总览与公共约定"，模块章节只引用。
- 原型中的视觉信息要落到前端可执行描述：组件、布局、交互、状态、校验，不只描述"按原型"。
- 对 PRD 与原型冲突的内容，单独列"冲突/待确认事项"。

## 输出风格

- 使用中文输出。
- 保持可直接复制进设计书的 Markdown 格式。
- 对长文档先输出完整目录和一个模块样例；用户确认后再继续补齐剩余模块。

- 字段字典、状态机等密集信息**必须用表格**，不要用列表或描述性段落。
