# Frontend Design Doc Generator v3.0.0

根据 PRD、原型/UI 设计、现有代码库和开发者补充信息，生成可直接指导开发的前端项目设计书。

## 版本标识（必须遵守）

本 skill 当前版本为 **v3.0.0**。生成的每份设计书**必须**在基础信息表格中包含以下追溯字段：

| 追溯字段 | 值 | 说明 |
| --- | --- | --- |
| 生成工具 | frontend-design-doc-generator | 固定值 |
| 工具版本 | v3.0.0 | 取本 skill 的 version |
| 生成时间 | 当前日期 | 设计书首次生成时间 |

如果输出中缺少这三个字段，说明**没有使用本版本的 skill**。

## 何时使用

- 用户提供 PRD、CoDesign/蓝湖/Figma/MasterGo/截图等原型或 UI 信息，需要输出前端设计书
- 用户要求把需求拆成前端模块、页面、路由、组件、接口和状态流转
- 用户已有设计书初稿，需要按统一结构补全或重写
- 用户要求对照现有项目技术栈、组件库、路由和 Store 模式生成开发方案

## ⛔ 结构化输出硬约束（不可违反）

> 本节为 v3.0.0 核心约束。以下规则优先级最高，任何其他指令不得覆盖。

### 量化底线

| 指标 | 最低值 | 检查方式 |
| --- | --- | --- |
| 每模块代码块总数 | ≥ 4 | 路由代码 + API Map + TypeScript interface + Hooks |
| 每模块 TypeScript interface | ≥ 2 | 请求参数类型 + 响应数据类型 |
| 每模块状态机表格 | ≥ 1 | 有状态流转时必须完整列出 |
| 每模块 ASCII 线框图 | ≥ 1 | 列表/详情/编辑页各一张 |
| 全文 Evidence 引用（代码） | ≥ 10 | 指向真实代码文件的 `Evidence(path:line)` |
| 全文 Evidence 引用（PRD） | ≥ 5 | 引用 PRD 章节/行号 |
| 全文 TODO 总数 | ≤ 20 | 超出说明代码库探索不充分 |
| 自检清单 | 15 项全列 | 末尾必须追加完整自检表 |

### 禁止项（违反即不合格）

| # | 禁止内容 | 替代方案 |
| --- | --- | --- |
| 1 | 禁止出现"按原型"、"按 PRD"、"参考设计稿"等不落地描述 | 必须转化为：组件、布局、交互、状态、校验的具体描述 |
| 2 | 禁止接口路径写 `TODO` | 必须基于项目现有路由/API 命名约定推导，可标 `(待确认)` |
| 3 | 禁止字段字典用列表或描述性段落 | 必须用 6 列表格：字段名 / key / 控件类型 / 必填 / 校验规则 / 枚举值 |
| 4 | 禁止省略任何 7 子章节之一 | 无内容时标注"本模块不涉及此项"并说明原因 |
| 5 | 禁止不写代码示例 | 每模块至少 4 类代码块（路由/API Map/Interface/Hooks） |
| 6 | 禁止编造不存在的文件路径、组件名、权限码 | 不确定的统一标 `(待确认)` |
| 7 | 禁止状态机用纯文字描述 | 必须用表格：状态 / UI 表现 / 允许动作 / 禁止动作 |
| 8 | 禁止遗漏 PRD 中的需求模块 | 必须逐一核对 PRD 需求列表，确保全覆盖 |

### 强制表格格式

以下表格在任何出现时**必须**包含指定列：

| 表格类型 | 必须包含的列 |
| --- | --- |
| 字段字典 | 字段名 \| key \| 控件类型 \| 必填 \| 校验规则 \| 枚举值/数据源 |
| 状态机 | 状态 \| UI 表现 \| 允许动作 \| 禁止动作 |
| 接口定义 | 接口名称 \| 方法 \| 路径 \| 入参 \| 出参 \| 调用时机 |
| 权限控制 | 权限层级 \| 控制对象 \| 规则 \| 无权限展示 |
| 组件复用 | 组件 \| 来源 \| 职责 \| props \| emit \| 使用场景 \| Evidence |
| 文件变更 | 类型 \| 操作 \| 路径 |
| 自检结果 | 检查项 \| 状态 \| 说明 |

## 输入材料

优先收集以下信息，不完整时也可以先产出草稿，并把缺失项标为 `(待确认)`：

| 材料 | 用途 | 缺失时处理 |
| --- | --- | --- |
| PRD 需求文档 | 业务逻辑、字段定义、权限矩阵、生命周期规则、审批规则 | 标注 `(待确认：PRD 未提供)` |
| 原型/UI 设计 | 页面布局、交互细节、组件样式、状态流转 | 标注 `(待确认：原型未提供)` |
| 现有代码库 | 技术栈、组件库、路由约定、Store 模式、接口封装方式 | 读取代码后对齐现有实现 |
| 模块划分 | 章节顺序、优先级、开发边界 | 先基于 PRD 自动拆分，再请用户确认 |
| 开发者补充 | 特殊约束、字段命名、接口约定、兼容逻辑 | 作为最高优先级输入 |

如果链接需要登录（如语雀、CoDesign、Confluence、私有 GitHub），不要使用 WebFetch 直接抓取；请让用户提供可访问内容、截图、导出的 Markdown/PDF，或在有浏览器/MCP 授权时通过对应工具读取。

### MasterGo 原型处理

如果用户提供 MasterGo 链接（含 `fileId`、`layerId`），**必须**使用 `mastergo-magic-mcp` 的 `getDsl` 工具读取设计稿 DSL 数据，从中提取：

- 页面宽度、布局方式（flex/grid/absolute）
- 组件层级关系
- 字体、颜色、间距等设计 token
- 交互状态（hover/active/disabled）

将 DSL 信息映射到前端组件和布局描述中，而不是仅写"按原型"。

## 工作流程

### 步骤 1：确认输入范围

1. 识别用户给出的 PRD、原型、UI、旧设计书、模块划分。
2. 如果用户只给链接且无法访问，说明限制并请求粘贴关键内容或提供本地文件。
3. 如果已有设计书初版，先复用其模块和业务信息，再按本 skill 的结构重排补齐。
4. **必须**逐一核对 PRD 需求列表，确保所有需求模块都被覆盖（包括迭代优化清单）。
5. 不要编造不存在的文件路径、组件名、权限码；不确定内容统一标 `(待确认)`。

### 步骤 2：代码库探索（强制执行，不可跳过）

**当项目目录是代码仓库时，本步骤为强制执行。**

必须使用 Agent（Explore 类型）或文件读取工具，扫描以下内容并记录发现：

| 探索项 | 必须读取的内容 | 产出 |
| --- | --- | --- |
| 技术栈 | `package.json`、`tsconfig.json`、构建配置 | 框架版本、UI 库、表格库、状态管理 |
| 目录结构 | `src/` 或 `pages/` 目录树 | 页面组织方式、公共组件位置 |
| 列表页模式 | 找到至少 2 个现有列表页的实现文件 | `createPage` / 自定义列表模式、分页字段 |
| 权限模式 | `hasAuth` / `useAuth` / `codes` 定义方式 | 权限码格式、按钮控制方式 |
| 请求封装 | `request/` 或 `api/` 目录 | APIMap 格式、接口注册方式 |
| 表单模式 | `useFormItem` / 表单项配置方式 | 控件类型、字典消费方式 |
| 相关模块 | 当前需求最相关的页面文件 | 挂载点、已有组件可复用项 |
| 上传/导入 | 现有上传组件、Excel 处理工具 | 可复用的上传/导入能力 |
| 路由配置 | 路由注册文件 | 路由定义格式、懒加载方式 |
| 类型定义 | 现有 typing.ts / types.ts | 类型命名约定 |

每个探索项**必须**在设计书中以 `Evidence(path:line)` 格式引用代码位置：

```
Evidence(`apps/pim/pages/preselected/index.tsx:216`)
```

**Evidence 最低要求**：代码引用 ≥ 10 处 + PRD 引用 ≥ 5 处 = 总计 ≥ 15 处。

### 步骤 3：模块拆分（混合模式）

采用**混合拆分模式**：

```
输出目录/
├── 00_总览与公共约定.md          ← 职能分层（架构+路由+权限+公共组件+技术栈）
├── 01_{模块A}_设计书.md          ← 业务模块平铺（7 子章节 + 强制代码块）
├── 02_{模块B}_设计书.md          ← 每个 PRD 功能模块一份独立文件
├── ...
├── N-1_系统集成与兼容.md         ← 横切关注点（新老系统兼容/消息推送等）
└── N_自检与测试策略.md           ← 质量保障（自检15项+测试要点+错误处理）
```

**拆分原则**：
- 总览文件（00）：放所有跨模块公共内容
- 业务模块文件（01~N-2）：按 PRD 功能域拆分，每模块含完整 7 子章节
- 横切文件（N-1）：系统集成、数据同步、消息通知等跨模块能力
- 质量保障文件（N）：自检清单 + 测试策略 + 错误处理规范

每个模块至少明确：

| 维度 | 要回答的问题 |
| --- | --- |
| 页面 | 有哪些列表、详情、新建、编辑、审批、弹窗、抽屉 |
| 数据 | 页面初始化、查询、提交、审批、导出、上传的链路 |
| 状态 | 草稿、申请中、已通过、已驳回、已撤销、待执行等状态如何影响 UI |
| 权限 | 按钮级、行级、数据级分别怎么控制 |
| 复用 | 复用哪些现有组件，哪些需要新建公共组件 |
| 风险 | 哪些字段、接口、规则需要后端/产品确认 |

### 步骤 4：按固定章节输出

每个业务模块**必须**包含以下 **7 个子章节**，格式参考 [输出模板](./references/output-template.md)：

1. **页面结构** — 路由代码 + 层级 + 布局 ASCII 图 + 交互
2. **数据链路** — 初始化 + 查询 + 提交 + 联动
3. **状态机与交互规则** — 状态枚举表 + 转换图 + 行级状态矩阵
4. **字段字典** — 完整 6 列表格（key / 控件 / 必填 / 校验 / 枚举）
5. **组件复用/沿用** — 复用表 + 新建组件 + Props 定义
6. **相关接口** — API Map 代码块 + TypeScript Interface + 接口表格
7. **补充逻辑 + Hooks** — 自定义 Hooks 代码块 + 权限 + 校验 + 性能

**每个模块必须包含的代码块**：

| 代码块类型 | 最低数量 | 说明 |
| --- | --- | --- |
| 路由配置代码 | 1 段 | 完整 `import()` 懒加载路由定义 |
| API Map 定义 | 1 段 | 遵循项目现有 `[method, path, ReqType, ResType]` 四元组格式 |
| TypeScript Interface | 2 段 | 请求参数 + 响应数据的完整类型定义 |
| 自定义 Hooks | 1 段 | 业务逻辑封装（价格计算/权限判断/状态管理/数据脱敏等） |
| ASCII 页面线框图 | 1 张 | 页面布局可视化 |
| 状态机表格 | 1 张 | 状态/UI 表现/允许动作/禁止动作 |

公共组件、公共权限、公共路由规范、公共数据字典放在"00_总览与公共约定"中，模块内只引用，不重复定义。

### 步骤 5：文件变更计划（强制执行）

**必须**在每个模块文件末尾输出文件变更计划表：

| 类型 | 操作 | 路径 |
| --- | --- | --- |
| 页面 | 修改 | 具体文件路径 — 修改内容简述 |
| 组件 | 新建 | 具体文件路径 |
| API | 修改 | 具体文件路径 — 新增接口 |
| 类型 | 新建 | 具体文件路径 — 新增类型 |

路径**必须**基于步骤 2 的代码库探索结果，不允许编造路径。

### 步骤 6：自检（强制执行）

输出完成后，**必须**在最后一个文件（N_自检与测试策略.md）中追加完整自检结果：

```markdown

| --- | --- | --- | --- | --- |

| 2 | 每个模块含 7 个子章节 | 完整性 | ✅/❌ | |
| 3 | 每模块代码块 ≥ 4 | 量化 | ✅/❌ | 平均 N 个 |
| 4 | 每模块 TypeScript interface ≥ 2 | 量化 | ✅/❌ | |
| 5 | 每模块状态机表格已定义 | 完整性 | ✅/❌ | |
| 6 | 每模块字段字典表格已列出 | 完整性 | ✅/❌ | |
```

## 代码示例要求

### 必须包含的代码示例类型

#### 1. 路由配置代码

必须遵循项目现有路由文件格式，完整写出 `import()` 懒加载定义：

```typescript
// 示例格式（需对齐项目实际格式）
['pim:moduleName:list', {
  path: '/moduleName/index',
  name: 'ModuleNameList',
  component: () => import('#/pim/pages/moduleName/index.tsx'),
  meta: { title: '模块名称', keepAlive: true }
}],
```

#### 2. API Map 定义

必须遵循项目现有 API Map 四元组格式：

```typescript
// 示例格式（需对齐项目实际格式）
'module_list': ['post', '/goods/module/page',
  T.ModulePageReq, T.ModulePageRes],
'module_detail': ['get', '/goods/module/detail',
  { id: string }, T.ModuleDetailRes],
```

#### 3. TypeScript Interface

请求参数和响应数据必须分别定义完整的 interface：

```typescript
interface ModulePageReq {
  current: number
  pageSize: number
  field1?: string
  field2?: string[]
}

interface ModulePageRes {
  total: number
  list: ModuleItem[]
}

interface ModuleItem {
  id: string
  name: string
  // ... 所有字段
}
```

#### 4. 自定义 Hooks

业务逻辑封装必须包含完整的函数签名和关键实现：

```typescript
export function useBusinessLogic(params: Ref<Params>) {
  const data = ref<Data>()

  // 核心业务逻辑
  const calculate = () => { ... }

  // 联动监听
  watch(() => params.value, () => { ... })

  return { data, calculate }
}
```

#### 5. ASCII 页面线框图

每个列表/详情/编辑页必须至少一张：

```text
┌──────────────────────────────────────────────┐
│ 筛选区：字段1 | 字段2 | 字段3   [查询][重置]  │
├──────────────────────────────────────────────┤
│ 操作栏：新建 / 导入 / 导出 / 批量操作         │
├──────────────────────────────────────────────┤
│ 表格区：列1 │ 列2 │ 列3 │ 状态 │ 操作        │
├──────────────────────────────────────────────┤
│ 分页栏：页码 / 每页条数 / 总数                │
└──────────────────────────────────────────────┘
```

## 编写规范

- **表格优先于长文本**，列表优先于大段叙述。
- **Evidence 引用**：引用代码库事实时**必须**带 `Evidence(path:line)` 格式；引用 PRD 事实时标注 `PRD:行号` 或章节名。
- 字段、接口、权限码不确定时不要编造，统一标 `(待确认)`。
- 路由详情页与编辑页**优先**复用同一路由，通过 `?id=xxx&mode=edit` 或 `?mode=view` 区分，除非项目已有不同约定。
- 列表页**必须**沿用项目现有列表构建方式；如果项目已有 `createPage`，明确写"使用 createPage 构建"并附 Evidence。
- 权限控制**必须**沿用项目现有方式；如果项目已有 `useAuth` / `hasAuth`，在按钮级、行级、数据级分别说明。
- 公共组件集中放在"00_总览与公共约定"，模块章节只引用。
- 原型中的视觉信息要落到前端可执行描述：组件、布局、交互、状态、校验，不只描述"按原型"。
- 对 PRD 与原型冲突的内容，单独列"冲突/待确认事项"。

## 输出风格

- 使用中文输出。
- 保持可直接复制进设计书的 Markdown 格式。
- 对长文档先输出完整目录和一个模块样例；用户确认后再继续补齐剩余模块。
- 如果用户明确要求一次性完整输出，则完整生成，不省略章节。
- 字段字典、状态机等密集信息**必须用表格**，不要用列表或描述性段落。
- 代码块必须标注语言类型（typescript / text / markdown）。

## 质量检查清单

输出前**必须**逐项检查：

- [ ] 基础信息表是否包含 `生成工具`、`工具版本 v3.0.0`、`生成时间` 三个追溯字段
- [ ] 每个模块是否都有 7 个固定子章节
- [ ] 每模块代码块是否 ≥ 4（路由/API Map/Interface/Hooks）
- [ ] 每模块 TypeScript interface 是否 ≥ 2（请求+响应）
- [ ] 每模块是否有状态机表格（有状态流转时）
- [ ] 每模块是否有字段字典表格（有表单/表格时）
- [ ] 每模块是否有 ASCII 线框图 ≥ 1
- [ ] 每个列表页是否有筛选区、操作区、表格列、分页/排序说明
- [ ] 每个详情/编辑/新建页是否有路由、表单字段、校验、保存/提交链路
- [ ] 每个导入/导出/上传是否有文件格式、触发时机、错误反馈
- [ ] 是否列出文件变更计划（具体文件路径 + 操作类型）
- [ ] 是否列出接口、字段、权限缺失的待确认项（≤ 20 处）
- [ ] 是否对齐现有技术栈和组件库，而不是输出通用方案
- [ ] Evidence 引用是否 ≥ 15 处（代码 ≥ 10 + PRD ≥ 5）
- [ ] PRD 需求列表是否全覆盖（含迭代优化清单）
- [ ] 自检结果 15 项是否已追加到文档末尾
- [ ] 无禁止项违反
