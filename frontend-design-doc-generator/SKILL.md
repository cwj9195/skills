# Frontend Design Doc Generator

根据 PRD、原型/UI 设计、API/Mock、现有代码库和开发者补充信息，生成 **AI 可独立实现的前端实施设计书**。

生成的设计书不是普通需求摘要，而是编码实施手册：一个未参与需求讨论的编码 AI 或开发者，只读取本文档和当前代码库，就应能开始并完成前端实现；无法实现的部分必须以结构化 TODO 标明阻塞影响、负责人和可选 mock/暂定方案。

## 核心目标：AI 可独立实现

设计书必须回答以下问题：

- 改哪些文件、新建哪些文件、放在哪些目录。
- 复用哪些现有页面、组件、API、Store、Hook、工具函数，路径和示例是什么。
- 页面、路由、菜单、权限、组件、状态、接口、校验如何落到代码。
- 筛选项、表格列、详情项、表单项的字段 key、类型、默认值、枚举、展示/编辑组件、校验规则是什么。
- API 的 method、path、request schema、response schema、错误处理、loading key、成功后刷新规则是什么。
- 新建组件的代码名、文件路径、props、emits、slots、exposed、状态来源和使用页面是什么。
- 实施顺序是什么，验收标准是什么，如何证明实现完成。
- 哪些信息缺失会阻塞实现，哪些可以暂定或 mock，后续替换点在哪里。

## 何时使用

- 用户提供 PRD、CoDesign/蓝湖/Figma/截图等原型或 UI 信息，需要输出前端设计书或实施设计书。
- 用户要求把需求拆成前端模块、页面、路由、组件、接口、状态流转和实施步骤。
- 用户已有设计书初稿，需要按统一结构补全、重写或升级为可编码实施文档。
- 用户要求对照现有项目技术栈、组件库、路由、API、权限和 Store 模式生成开发方案。
- 用户明确要求“让 AI 根据设计书直接开发”“AI 独立完成前端项目”“implementation-ready”。

## 输入材料

优先收集以下信息。不完整时可以先产出草稿，但必须把缺失项写入结构化 TODO，并标明是否阻塞实现：

| 材料 | 用途 | 缺失时处理 |
| --- | --- | --- |
| PRD 需求文档 | 业务逻辑、字段定义、权限矩阵、生命周期规则、审批规则 | 标注 `TODO：待 PRD 确认`，并说明影响模块 |
| PRD 本地文件 | 支持 Markdown/PDF 等本地文件作为一等输入源；优先读取业务正文、功能详细说明、需求列表和术语表 | 文件不存在时标注 `TODO：待提供 PRD 文件` |
| 原型/UI 设计 | 页面布局、交互细节、组件样式、状态流转 | 标注 `TODO：待原型确认`；不得写“按原型”替代实现描述 |
| MasterGo/Figma/CoDesign 图层 | 设计节点、组件层级、设计 token、交互状态 | 记录读取状态；缺图层 ID 时标注阻塞 UI 还原 |
| 目标项目根目录 | 定位代码约定、路由/API/组件/Store 文件落点 | 如果当前目录是项目则使用当前代码库；否则标注 `TODO：待确认项目根目录` |
| 现有代码库 | 技术栈、组件库、路由约定、Store 模式、接口封装方式、测试方式 | 读取代码后对齐现有实现；未确认写 `TODO：待代码库确认` |
| API 文档 / Mock / OpenAPI | method、path、request、response、error、分页、上传下载协议 | 缺失时 API 作为 P0 TODO；如可先 mock，必须写 mock contract |
| 设计系统 / 组件库文档 | 组件行为、表单/表格规范、token、样式状态 | 优先从代码库和组件文档确认；否则标注 TODO |
| 代码入口约定 | 路由注册、菜单注册、权限注册、API/type/store 目录 | 未确认时不得编造路径，需列入阻塞或待确认 |
| 模块划分 | 章节顺序、优先级、开发边界 | 先基于 PRD 自动拆分，再请用户确认或标注依据 |
| 开发者补充 | 特殊约束、字段命名、接口约定、兼容逻辑 | 作为最高优先级输入 |

如果链接需要登录（如语雀、CoDesign、Confluence、私有 GitHub），不要使用 WebFetch 直接抓取；请让用户提供可访问内容、截图、导出的 Markdown/PDF，或在有浏览器/MCP 授权时通过对应工具读取。

## 证据与 TODO 规则

1. 设计书必须区分“已确认事实”“推断建议”“TODO/暂定”。
2. 来自代码库的约定必须标注文件路径、符号名或现有示例；未读取代码不得写成事实。
3. `TODO` 不能替代整个章节；即使缺少真实接口，也要给出 mock contract 或说明该步骤被阻塞。
4. `暂定` 只能用于结构清晰但名称、字段 key、路径、权限码未最终确认的内容。
5. 每个 TODO 必须尽量结构化：

| ID | 类型 | 问题 | 是否阻塞实现 | 影响范围 | 建议默认方案 | 负责人 |
| --- | --- | --- | --- | --- | --- | --- |
| T001 | API/字段/权限/原型/代码库 | TODO | 是/否 | TODO | TODO/mock/暂定 | 产品/后端/前端/设计 |

## PRD 本地文件读取规则

1. PRD 可以是链接，也可以是本地 Markdown/PDF 文件；本地 `.md` 文件视为最高可读证据，必须优先读取原文而不是让用户重复粘贴。
2. 读取 PRD 时先定位“功能详细说明”“需求实现逻辑”“需求列表”“名词释义”等业务正文区域，并保留路径与标题锚点作为证据。
3. 如果 PRD 标题层级不稳定，不硬绑定三级标题；标题层级只作为辅助证据，最终按业务闭环识别模块。
4. PRD 中的图片、附件、外链只作为线索；无法读取的图片内容不得臆造，需标注 `TODO：待补充图片/附件内容`，并说明是否影响 UI 还原或字段确认。
5. 对本地 PRD 批量生成设计书时，先提取公共背景、术语、权限、需求列表，再按模块分别生成，避免跨模块信息互相污染。

## MasterGo MCP 读取规则

当输入材料包含 MasterGo 设计稿链接时，优先使用 `mastergo-magic-mcp`，不要退化为截图猜测。PRD、原型或 UI 说明中只要出现 `mastergo.com` 链接，就必须先按本规则处理，再生成设计书。读取顺序如下：

1. 先解析链接参数：`fileId` 取 `/file/<fileId>`；目标图层只接受 `layer_id` 或 `source_layer_id`；`page_id` 只表示页面，不等同于图层 ID。
2. 如果链接是短链或包含 `layer_id`，优先调用 `mcp__mastergo_magic_mcp__.mcp__getDsl` 获取 DSL；需要页面级站点信息时，再调用 `mcp__getMeta`。
3. 如果 `getDsl` 返回 `componentDocumentLinks` 且非空，必须逐个调用 `mcp__getComponentLink` 获取组件文档，再结合 DSL 生成设计书。
4. 如果链接只有 `page_id`，不得把 `page_id` 当作 `layerId` 下结论；应标记为 `TODO：待提供带 layer_id 的 MasterGo 图层链接`，或在有浏览器授权时通过 Chrome/Computer Use 辅助用户复制“选中图层链接”。
5. 如果 MasterGo MCP 返回空 `nodes/components/styles`，按证据记录为空结果，不能据此臆造页面结构；需要请求具体画板、容器或组件的 `layer_id`。
6. 生成设计书时，视觉信息必须来自 DSL、组件文档、PRD 或用户补充；缺失字段、接口、权限码统一进入结构化 TODO 或标 `暂定`。
7. 视觉信息必须转成可执行前端描述：布局区域、组件、状态、交互、样式 token、响应式规则、loading/empty/error/disabled 状态，而不是只写“按设计稿”。

## 工作流程

### 步骤 1：确认输入范围与访问能力

1. 识别用户给出的 PRD、原型、UI、API 文档、旧设计书、模块划分、项目根目录。
2. 如果用户只给链接且无法访问，说明限制并请求粘贴关键内容或提供本地文件。
3. 如果已有设计书初版，先复用其模块和业务信息，再按本 skill 的实施结构重排补齐。
4. 不要臆造不存在的接口路径、字段 key、权限码；不确定内容统一标 `暂定` 或结构化 `TODO`。
5. 如输入包含 MasterGo 链接，先按“MasterGo MCP 读取规则”完成 DSL/Meta/组件文档读取；无法取得 `layer_id` 时先记录 TODO，不进入细节推断。

### 步骤 2：读取证据并提取事实

1. 从 PRD、原型、MasterGo、API 文档、代码库和开发者补充中提取事实。
2. 为每个模块记录证据来源：路径/链接、章节、图层、代码文件、读取状态。
3. 把无法确认但影响实现的内容列为 TODO，并标注阻塞影响。
4. 对 PRD 与原型、API、代码库冲突的内容，单独列“冲突/待确认事项”。

### 步骤 3：读取现有代码库约定

如用户要求“贴合项目代码”或当前目录是代码库，优先检查并输出“代码库证据清单”：

| 维度 | 需要确认 | 输出要求 |
| --- | --- | --- |
| 技术栈 | Vue/React、Vite/Webpack、TypeScript、状态管理 | 标注 package 或配置文件路径 |
| 目录约定 | pages/views、components、api/service、types、store/hooks、styles、tests | 标注现有目录和示例文件 |
| UI 组件库 | 表单、表格、弹窗、抽屉、上传、字典组件 | 标注库名、import 示例、现有使用文件 |
| 路由/菜单 | 路由注册、菜单入口、详情/编辑/审批路由复用 | 标注注册文件和类似模块 |
| 列表页封装 | 如 `createPage`、分页字段、查询按钮、表格配置方式 | 标注封装来源和示例 |
| 权限控制 | 如 `useAuth`、`hasAuth`、权限码组织方式 | 标注工具路径、按钮/行级示例 |
| 请求封装 | 接口目录、API 方法命名、响应包裹、错误处理 | 标注 request wrapper 和 API 示例 |
| Store/缓存 | Pinia/Vuex/Zustand/Redux、是否持久化 | 标注 store 示例和清理时机 |
| 字典/格式化 | 字典来源、日期/金额/状态格式化 | 标注工具或组件示例 |
| 上传/下载 | 文件上传、导入、导出、下载中心 | 标注已有能力和调用方式 |
| 样式方案 | Less/Sass/CSS modules/Tailwind/CSS-in-JS、token | 标注样式文件和命名习惯 |
| 测试方式 | unit/component/e2e、命令、mock 方式 | 标注配置和示例 |

输出时只能引用已确认的现有约定；未确认时写成“建议”或 `TODO：待代码库确认`。

### 步骤 4：模块拆分与实现范围确认

按业务闭环拆模块，而不是按接口或组件拆模块。模块识别优先级如下：

1. 用户显式指定的模块划分。
2. PRD“功能详细说明”等正文区域中的语义模块。
3. 需求列表表格中的模块/需求目标。
4. 标题层级、页面入口、申请单闭环、权限块和 MasterGo 链接聚合关系。

类似“一期迭代优化清单”这类非标准标题，不写死为固定规则；应识别为“横向优化/公共优化类模块”，并根据 PRD 当前内容决定是否独立成文。每轮生成前先产出模块清单，包含编号、模块名、来源段落、关联 MasterGo 链接、实现范围、输出文件名。

每个模块至少明确：

| 维度 | 要回答的问题 |
| --- | --- |
| 页面 | 有哪些列表、详情、新建、编辑、审批、弹窗、抽屉；哪些不在本期范围 |
| 文件 | 预计新建/修改/复用哪些页面、组件、API、类型、Store/Hook、样式、测试文件 |
| 数据 | 页面初始化、查询、提交、审批、导出、上传的链路 |
| 字段 | 筛选、表格、表单、详情、导入/导出的字段字典和 API 映射 |
| 状态 | 草稿、申请中、已通过、已驳回、已撤销、待执行等状态如何影响 UI |
| 权限 | 按钮级、行级、数据级分别怎么控制；权限码是什么或是否待确认 |
| 复用 | 复用哪些现有组件，哪些需要新建公共组件 |
| 风险 | 哪些字段、接口、规则、图层、权限需要后端/产品/设计确认 |

### 步骤 5：生成实施蓝图

每个模块在进入详细章节前，必须先生成实施蓝图：

- 文件变更清单：新建/修改/复用的页面、组件、API、类型、Store/Hook、样式、测试文件。
- 依赖顺序：先类型/API，再 Store/hooks，再组件，再页面，再路由/权限，再测试。
- 复用映射：需求中的页面/组件分别对应代码库中的哪些现有模式。
- 阻塞项：没有接口、字段、权限码、设计节点时，标明是否阻塞实现。
- mock 策略：后端接口缺失但可先开发时，给出 mock schema、mock 数据位置和替换点。

### 步骤 6：按实施章节输出

批量生成时，每个模块按同一 skill 流程独立生成一份文档，避免一个超长文档互相污染。输出文件名固定为 `NN_模块名_设计方案.md`，`NN` 从 01 开始递增，模块名需去除空格、斜杠等非法路径字符。

公共组件、公共权限、公共路由规范、公共数据字典可在每份文档中保留“公共约定引用”，但不要把无关模块的大段规则复制到当前模块。

每个业务模块必须包含以下实施级子章节，格式参考 [输出模板](./references/output-template.md)：

1. 模块证据与范围
2. 文件变更计划
3. 页面结构与路由
4. UI 布局与交互契约
5. 数据模型与字段字典
6. API 契约
7. 组件复用与新建组件契约
8. 状态、缓存与数据链路
9. 权限、校验与状态机
10. 异常、边界与性能
11. 实现步骤
12. 验收标准
13. TODO / 待确认 / 风险

公共组件、公共权限、公共路由规范、公共数据字典放在“公共约定”章节，模块内只引用，不重复长篇定义。

### 步骤 7：执行实施就绪质量检查

输出前必须检查：文档是否能让未参与需求讨论的 AI 只凭本文档和代码库开始编码；如果不能，必须明确列出阻塞 TODO，而不是用泛化描述掩盖缺口。

## 输出要求

### 文档结构

推荐顶层结构：

1. 基础信息
2. 证据与输入材料
3. 版本履历
4. 需求概述
5. 前端模块总览
6. 前端实现范围
7. 公共约定
8. 业务模块设计（逐模块）
9. 路由汇总
10. 接口清单
11. TODO / 待确认 / 风险
12. 验证计划

### 每个模块必须覆盖

#### 1. 模块证据与范围

必须包含：

- PRD、原型、MasterGo、API、代码库、开发者补充的证据来源。
- 当前模块 in scope / out of scope / future scope。
- 已确认事实、推断建议、关键假设、阻塞 TODO。

#### 2. 文件变更计划

必须包含：

| 类型 | 操作 | 建议文件路径 | 代码职责 | 复用参考 | 备注 |
| --- | --- | --- | --- | --- | --- |
| 页面/API/组件/类型/Store/Hook/样式/测试 | 新建/修改/复用 | TODO | TODO | TODO | TODO |

文件路径未知时，不得编造为事实；写 `TODO：待代码库确认`，并给出查找策略或参考目录。

#### 3. 页面结构与路由

必须包含：

- 路由定义：路径、query/path 参数、同一路由复用情况、页面文件、路由注册位置、权限码。
- 页面层级关系：列表 → 详情 → 弹窗/抽屉/审批区，用树形结构表示。
- 布局逻辑：用 ASCII 图表示筛选区、操作栏、表格区、分页栏、详情区。
- 元素排布：筛选项、操作按钮、表格列、底部操作区的位置关系。
- 页面状态：每个区域的 loading、empty、error、disabled、readonly 处理。
- 交互细节：悬停、点击、拖拽、键盘操作、右键菜单、二次确认、未保存拦截。

#### 4. UI 布局与交互契约

必须包含：

- 原型/DSL 节点与页面区域的映射。
- 组件层级、抽屉/弹窗尺寸、按钮位置、表单布局、表格滚动/固定列。
- loading/empty/error/disabled/readonly 状态文案。
- 交互流程、确认弹窗文案、成功/失败提示。
- PRD 与原型冲突时的处理结论或 TODO。

#### 5. 数据模型与字段字典

必须包含字段字典表：

| 字段中文名 | 字段 key | 类型 | 来源 | 默认值 | 是否必填 | 枚举/字典 | 展示组件 | 编辑组件 | 校验 | API 映射 | 备注 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |

同时必须说明：

- 列表项类型、详情类型、表单模型、查询参数、提交 payload、接口 response。
- UI 字段与 API 字段的映射规则。
- 日期、金额、数量、状态、人员、组织、附件、图片等特殊字段格式化规则。

#### 6. API 契约

必须包含接口表格：

| 接口名称 | API 函数名 | API 文件 | 方法 | 路径 | 入参 schema | 出参 schema | 错误处理 | 调用时机 | loading key | 成功后处理 | 状态 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |

如果接口未定：

- 路径写 `TODO：待后端确认`。
- 字段 key 未定，标注 `暂定`。
- 如前端可先开发，必须给 mock request/response schema。
- 如无法开发，必须标注 P0 阻塞和影响的实现步骤。

#### 7. 组件复用与新建组件契约

复用组件必须包含：

| 组件 | 来源路径/库 | 现有示例 | 使用方式 | 需要适配点 |
| --- | --- | --- | --- | --- |

新建组件必须包含：

| 中文组件名 | 代码名 | 建议文件路径 | 职责 | props | emits | slots | exposed | 状态来源 | 使用场景 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |

组件名称需有中文业务名；同时必须给出代码名或文件名建议，不能只写 `<业务表格 />`。

#### 8. 状态、缓存与数据链路

必须包含：

- 页面初始化加载哪些接口。
- 查询/筛选/分页/排序的请求时机和参数组装。
- 新建/编辑/审批/撤销/导入/导出的请求链路。
- 状态 owner：local state、route query、Store、server、缓存。
- Store 缓存字段、是否持久化、何时写入、何时清空。
- 数据变更后的联动刷新逻辑。
- 请求取消、防抖、重复提交、错误恢复策略。

#### 9. 权限、校验与状态机

必须包含：

- 权限矩阵：菜单、按钮、行操作、字段、数据范围、导出/上传/审批权限。
- 权限码：已确认写具体值；未确认写结构化 TODO。
- 无权限表现：隐藏、禁用、tooltip、后端兜底。
- 表单校验矩阵：字段、规则、触发时机、错误提示。
- 状态机：状态值、后端枚举、展示标签、允许操作、下一状态、按钮可见性。

#### 10. 异常、边界与性能

必须包含：

- 边界条件：空数据、超长文本、重复提交、并发操作、接口失败。
- 文件上传/导入/导出：格式、大小、数量、失败反馈、异步任务状态。
- 性能考量：虚拟滚动、防抖节流、懒加载、批量操作分片。
- 可访问性、键盘交互、埋点/日志，如项目已有相关要求。

#### 11. 实现步骤

必须包含：

| 顺序 | 步骤 | 涉及文件 | 前置依赖 | 完成标准 |
| --- | --- | --- | --- | --- |

实施顺序应可执行，例如：类型/API → Store/Hook → 组件 → 页面 → 路由/权限 → 测试。

#### 12. 验收标准

必须包含 Given/When/Then 形式的验收表：

| 场景 | Given | When | Then | 验证方式 |
| --- | --- | --- | --- | --- |

验收标准要具体到可人工或自动验证，不能只写“查询正常”“按权限展示”。

#### 13. TODO / 待确认 / 风险

必须使用结构化 TODO 表，且 P0 阻塞项必须进入全局 TODO 汇总。

## 编写规范

- 表格优先于长文本，列表优先于大段叙述。
- 字段、接口、权限码不确定时不要编造，统一标 `暂定` 或结构化 `TODO`。
- 路由详情页与编辑页优先复用同一路由，通过 `?id=xxx&mode=edit` 或 `?mode=view` 区分，除非项目已有不同约定。
- 列表页优先沿用项目现有列表构建方式；如果项目已有 `createPage`，明确写“使用 createPage 构建”，并标注来源文件。
- 权限控制优先沿用项目现有方式；如果项目已有 `useAuth` / `hasAuth`，在按钮级、行级、数据级分别说明，并标注工具路径。
- 公共组件集中放在“公共约定”，模块章节只引用。
- 原型中的视觉信息要落到前端可执行描述：组件、布局、交互、状态、校验，不只描述“按原型”。
- 对 PRD 与原型/API/代码库冲突的内容，单独列“冲突/待确认事项”。
- 查询型模块不要生成无依据的新建/审批；后台能力不要强行生成列表/详情/保存接口；通知/推送能力要明确前端是否有页面、状态展示或完全由后端处理。

## 质量检查清单

输出前检查：

- 是否有“证据与输入材料”表，并区分已确认、推断、TODO。
- 每个模块是否都有实施级子章节，而不是只有 5 个泛化章节。
- 是否列出每个模块的新建/修改/复用文件路径。
- 是否列出每个页面的路由、参数、页面文件、注册位置、权限码。
- 每个列表页是否有筛选区、操作区、表格列、分页/排序说明、loading/empty/error 状态。
- 每个详情/编辑/新建页是否有路由、表单字段、校验、保存/提交链路。
- 每个审批页是否有状态、审批人、操作按钮、通过/驳回链路。
- 每个导入/导出/上传是否有文件格式、触发时机、错误反馈、异步任务状态。
- 是否列出每个字段的 key、类型、默认值、校验、展示/编辑组件、API 映射。
- 是否列出每个接口的 API 函数名、文件路径、请求/响应 schema、错误处理、成功后处理。
- 是否列出每个新组件的 props、emits、slots、文件路径、使用方和状态来源。
- 是否列出状态 owner、缓存、刷新、错误处理规则。
- 是否列出权限码、权限矩阵、无权限展示方式；缺失时是否标为阻塞 TODO。
- 是否列出实现步骤和 Given/When/Then 验收标准。
- 是否列出接口、字段、权限、原型缺失的 TODO，并标明是否阻塞实现。
- 是否对齐现有技术栈和组件库，而不是输出通用方案。
- 如果使用 MasterGo，是否记录 `fileId`、`layer_id/source_layer_id`、MCP 返回状态，并避免把 `page_id` 当作图层 ID。
- 是否仍存在“按原型”“按需求”“参考现有”“后续确认”这类没有转成可执行行为的描述；如有，必须补成具体规则或结构化 TODO。
- 未参与需求讨论的 AI 是否可以只凭本文档和代码库开始编码；如果不能，阻塞点是否清晰可追踪。

## 输出风格

- 使用中文输出。
- 保持可直接复制进设计书的 Markdown 格式。
- 对长文档可以分批输出，但必须保持完整实施结构；如果用户要求“AI 独立实现 / 可直接开发 / 完整实施设计书”，不要只输出一个模块样例后停止。
- 如果用户明确要求一次性完整输出，则完整生成，不省略章节。
- 输出内容应避免泛化套话；每个结论尽量落到字段、文件、接口、组件、状态、权限、验收标准。
