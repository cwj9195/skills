# Frontend Design Doc Generator v6.0.0

根据 PRD、原型/UI 设计、现有代码库和开发者补充信息，交互式生成 **提纲确认版** 与 **AI 可独立实现的前端实施设计书**。

## 版本标识

| 追溯字段 | 值                            | 说明                  |
| -------- | ----------------------------- | --------------------- |
| 生成工具 | frontend-design-doc-generator | 固定值                |
| 工具版本 | v6.0.0                        | 取本 skill 的 version |
| 生成时间 | 当前日期                      | 设计书首次生成时间    |

## 交互式两阶段模式

本 skill 默认采用两阶段生成：先确认结构，再生成可开发版本。除非用户明确要求"直接出完整设计书 / 可开发版 / AI 落地版"，否则只提供 PRD 或原型时必须先进入 Outline Mode。

| 模式 | 目标 | 输入 | 输出 | 完成条件 |
| ---- | ---- | ---- | ---- | -------- |
| Outline Mode / 提纲版 | 快速确认需求拆分是否正确 | PRD、原型、旧设计书、补充说明 | 需求摘要、模块拆分、页面清单、关键流程、字段/API/权限初步假设、待确认问题 | 用户确认模块边界、优先级和关键假设 |
| Implementation Mode / AI 可落地开发版 | 生成可直接交给 AI/开发者实现的设计书 | 已确认提纲 + PRD + 原型 + 代码库 Evidence | `00_总览与公共约定.md`、业务模块设计书、系统集成与测试策略 | 满足 13 章节、Evidence、代码块、自检与 TODO 管理要求 |

### PRD 到 CI 链路输入输出

当本 skill 被 `FrontendSpecAgent` 调用时，必须继承以下链路，不允许只产出孤立设计书：

```text
PRD / 原型 / 旧设计书
  -> requirements.md
  -> facts.md / fact-set.md
  -> spec.md / design-outline.md
  -> design.md（本 skill 负责）
  -> tasks.md / claude-code-task.md
  -> ci-gate.md
```

Implementation Mode 的 `design.md` 必须接收 `facts.md` 或 `fact-set.md` 作为代码库事实输入；若缺失，先回到 Facts Agent 补证据。输出必须能被 Claude Code 直接消费：章节、接口、字段、组件、状态机、验收标准和 TODO 均应可追溯到 requirements/facts。

新增引用模板按需读取：

- `references/fact-set-template.md`：生成事实快照与 Evidence 表。
- `references/claude-code-task-template.md`：生成 Claude Code 任务包。
- `references/ci-gate-template.md`：生成本地与 CI 门禁清单。

### Agent 状态机

| 状态 | 触发条件 | Agent 行为 | 下一状态 |
| ---- | -------- | ---------- | -------- |
| `initial` | 收到 PRD、原型、旧设计书或模块说明 | 识别输入材料，判断是否已有提纲或明确要求完整设计书 | `outline` / `implementation` |
| `outline` | 用户只给 PRD/原型，或要求先拆模块 | 生成提纲版，不展开所有实现细节 | `confirm` |
| `confirm` | 提纲版输出完成 | 列出待确认问题，等待用户确认或补充 | `implementation` / `outline` |
| `implementation` | 用户确认提纲，或明确要求可开发版 | 基于确认提纲和代码库 Evidence 生成完整设计书 | 完成 / `repair` |
| `repair` | Evidence 不足、PRD 冲突、TODO 过多或关键约束不满足 | 回到确认问题，指出阻塞项和临时方案 | `confirm` |

### 模式选择规则

| 用户输入 | 默认模式 | 处理规则 |
| -------- | -------- | -------- |
| 只提供 PRD、原型或截图 | Outline Mode | 先输出提纲确认版，不强制完整 13 章节 |
| 提供上一轮提纲并要求细化 | Implementation Mode | 先核对提纲变更，再生成可落地开发版 |
| 明确说"完整设计书"、"可开发版"、"AI 落地版" | Implementation Mode | 直接执行完整 Evidence 探索和 v6 落地模板 |
| 已有旧设计书需要升级 | 根据用户意图判断 | 若未明确要求完整落地，先输出差异提纲 |

### Outline Mode 轻量硬约束

提纲版不是最终设计书，不要求完整 13 章节、6 类代码块和完整 Evidence 数量，但必须满足以下内容：

| # | 必须输出 | 要求 |
| - | -------- | ---- |
| 1 | 需求摘要 | 说明业务目标、范围、输入来源和缺失材料 |
| 2 | 模块拆分表 | 列出模块、页面、核心能力、优先级、依赖和风险 |
| 3 | 页面清单 | 列出页面/弹窗/抽屉、入口、主要交互和复用方向 |
| 4 | 关键流程 | 用步骤或 ASCII 流程表达主链路 |
| 5 | 字段/API/权限初步假设 | 只列候选项和确认点，不编造最终契约 |
| 6 | 二阶段补齐清单 | 声明进入 Implementation Mode 后补齐的 Evidence、代码块、字段字典、API 契约、状态机和验收标准 |
| 7 | 待确认问题 | 以结构化表格列出问题、影响、推荐默认值和确认后修改范围 |
| 8 | 提纲自检 | 检查模块覆盖、页面覆盖、关键 TODO、是否可进入第二阶段 |

### MCP 与探索规则

| 场景 | 优先工具 | 规则 |
| ---- | -------- | ---- |
| 代码结构、符号、调用关系、依赖、影响面 | Codegraph MCP | 必须优先使用 `codegraph_*`；不足时再降级读取文件 |
| 链接、页面、登录态站点、浏览器可见信息 | Chrome DevTools MCP | 链接访问优先通过 Chrome DevTools MCP |
| MasterGo 原型 | `mastergo-magic-mcp.getDsl` | 提取 DSL 后映射为布局、组件、token 与交互状态 |
| Outline Mode | Codegraph + 必要文件读取 | 只做必要探索，避免过度读取 |
| Implementation Mode | Codegraph + 文件读取 + 原型工具 | 必须完成完整 Evidence 探索，并满足代码 Evidence 与 PRD Evidence 数量要求 |

## ⛔ Implementation Mode 硬约束声明（最高优先级，不可违反）

> 以下规则仅强制作用于 Implementation Mode / AI 可落地开发版。Outline Mode 使用上方轻量硬约束。

### A. 章节标题锁定

每个业务模块**必须且只能**包含以下 **13 个章节**，标题为精确文字：

**禁止修改、禁止自拟、禁止合并、禁止调换顺序、禁止省略。**

无内容时标注"本模块不涉及此项"并说明原因，但**章节标题必须保留**。

| 序号 | 章节标题（精确文字，禁止修改）     |
| ---- | -------------------------------- |
| 1    | 模块证据与范围                   |
| 2    | 文件变更计划                     |
| 3    | 页面结构与路由                   |
| 4    | UI 布局与交互契约                |
| 5    | 数据模型与字段字典               |
| 6    | API 契约                         |
| 7    | 组件复用与新建组件契约           |
| 8    | 状态、缓存与数据链路             |
| 9    | 权限、校验与状态机               |
| 10   | 异常、边界与性能                 |
| 11   | Hooks 与工具函数                 |
| 12   | 实现步骤                         |
| 13   | 验收标准与 TODO                  |

### B. 表格列名锁定

以下表格在任何出现时**必须**包含指定列，列名和列数禁止修改：

| 表格类型       | 列数 | 必须包含的列                                                                                                                                  |
| -------------- | ---- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| 代码库证据清单 | 6    | 探索项 \| 文件路径 \| 发现结论 \| 复用方式 \| import/注册位置 \| 可信度                                                                       |
| 实施蓝图       | 5    | 变更类型 \| 文件路径 \| 实现内容 \| 依赖 \| 参考实现                                                                                          |
| 字段字典       | 10   | 字段标签 \| 字段 key \| 类型 \| 控件类型 \| 必填 \| 默认值 \| 校验规则 \| 枚举值/数据源 \| API 映射 \| 展示规则                               |
| API 契约       | 12   | 接口名称 \| API 函数名 \| API 文件 \| 方法 \| 路径 \| 入参 schema \| 出参 schema \| 错误处理 \| 调用时机 \| loading key \| 成功后处理 \| 状态 |
| 状态机         | 5    | 状态 \| 中文名 \| UI 表现 \| 允许动作 \| 禁止动作                                                                                             |
| 权限控制       | 4    | 权限层级 \| 控制对象 \| 规则 \| 无权限展示                                                                                                    |
| 复用组件       | 8    | 组件 \| 来源路径 \| 现有示例 \| 职责 \| props \| emit/slots \| 使用场景 \| Evidence                                                           |
| 新建组件契约   | 9    | 中文名 \| 代码名 \| 文件路径 \| 职责 \| props \| emits \| slots \| exposed \| 使用方                                                          |
| 文件变更       | 5    | 变更类型 \| 文件路径 \| 实现内容 \| 依赖 \| 参考实现                                                                                          |
| 状态与缓存     | 6    | 状态名 \| 存放位置 \| 初始值 \| 写入时机 \| 清空时机 \| 是否持久化                                                                            |
| 实现步骤       | 5    | 步骤 \| 操作 \| 涉及文件 \| 前置依赖 \| 完成标准                                                                                              |
| 验收标准       | 5    | 场景 \| Given \| When \| Then \| 验证方式                                                                                                     |
| TODO 清单      | 9    | ID \| 类型 \| 问题 \| 是否阻塞实现 \| 影响范围 \| 临时方案 \| 负责人 \| 优先级 \| 确认后需修改章节                                            |
| 自检结果       | 4    | # \| 检查项 \| 状态 \| 说明                                                                                                                   |

### C. 禁止项（违反即不合格）

| #   | 禁止内容                                             | 替代方案                                                         |
| --- | ---------------------------------------------------- | ---------------------------------------------------------------- |
| 1   | 禁止出现"按原型"、"按 PRD"、"参考设计稿"等不落地描述 | 必须转化为：组件、布局、交互、状态、校验的具体描述               |
| 2   | 禁止接口路径写 `TODO`                                | 必须基于项目现有路由/API 命名约定推导，可标 `(待确认)`           |
| 3   | 禁止字段字典用列表或描述性段落                       | 必须用 10 列表格                                                 |
| 4   | 禁止省略任何 13 子章节之一                           | 无内容时标注"本模块不涉及此项"并说明原因                         |
| 5   | 禁止不写代码示例                                     | 每模块至少 6 类代码块（路由/API Map/Interface×2/Hooks/组件契约） |
| 6   | 禁止编造不存在的文件路径、组件名、权限码             | 不确定的统一标 `(待确认)`                                        |
| 7   | 禁止状态机用纯文字描述                               | 必须用表格                                                       |
| 8   | 禁止遗漏 PRD 中的需求模块                            | 必须逐一核对 PRD 需求列表，确保全覆盖                            |
| 9   | 禁止 TODO 替代整个章节                               | TODO 只能用于具体字段/路径/码值，不能替代章节内容                |
| 10  | 禁止验收标准只写"功能正常"                           | 必须用 Given/When/Then 格式描述具体场景                          |

### D. 量化底线

| 指标                        | 最低值          | 说明                                                               |
| --------------------------- | --------------- | ------------------------------------------------------------------ |
| 每模块代码块总数            | ≥ 6             | 路由代码 + API Map + TypeScript interface×2 + Hooks + 组件契约     |
| 每模块 TypeScript interface | ≥ 2             | 请求参数类型 + 响应数据类型                                        |
| 每模块状态机表格            | ≥ 1             | 有状态流转时必须完整列出                                           |
| 每模块 ASCII 线框图         | ≥ 1             | 列表/详情/编辑页各一张                                             |
| 每模块实施蓝图              | = 1             | 每个模块必须包含文件变更清单、依赖顺序、阻塞项                     |
| 每模块验收标准              | ≥ 3             | 至少 3 条 Given/When/Then 场景                                     |
| 代码库证据清单              | ≥ 1             | 步骤 2 必须输出结构化证据清单                                      |
| 全文 Evidence 引用（代码）  | ≥ 15            | 指向真实代码文件的 `Evidence(path:line)`                           |
| 全文 Evidence 引用（PRD）   | ≥ 8             | 引用 PRD 章节/行号                                                 |
| 全文 TODO 总数              | ≤ 15            | 超出说明代码库探索不充分                                           |
| 阻塞型 TODO                 | 100% 有临时方案 | P0 TODO 必须含 mock contract 或建议默认方案                        |

## 何时使用

- 用户提供 PRD、CoDesign/蓝湖/Figma/MasterGo/截图等原型或 UI 信息，需要输出前端设计书
- 用户要求把需求拆成前端模块、页面、路由、组件、接口和状态流转
- 用户已有设计书初稿，需要按统一结构补全或重写
- 用户要求对照现有项目技术栈、组件库、路由和 Store 模式生成开发方案

## 输入材料

| 材料                  | 用途                                                 | 缺失时处理                                        |
| --------------------- | ---------------------------------------------------- | ------------------------------------------------- |
| PRD 需求文档          | 业务逻辑、字段定义、权限矩阵、生命周期规则、审批规则 | 标注 `(待确认：PRD 未提供)`                       |
| 原型/UI 设计          | 页面布局、交互细节、组件样式、状态流转               | 标注 `(待确认：原型未提供)`                       |
| 现有代码库            | 技术栈、组件库、路由约定、Store 模式、接口封装方式   | 读取代码后对齐现有实现                            |
| API 文档/Mock/OpenAPI | 接口 method/path/request/response/error              | 标注 `(待确认：API 文档未提供)`，基于命名约定推导 |
| 设计系统/组件库文档   | 设计 token、组件行为、表单/表格规范                  | 读取代码中已有用法推导                            |
| 模块划分              | 章节顺序、优先级、开发边界                           | 先基于 PRD 自动拆分，再请用户确认                 |
| 开发者补充            | 特殊约束、字段命名、接口约定、兼容逻辑               | 作为最高优先级输入                                |

如果链接需要登录（如语雀、CoDesign、Confluence、私有 GitHub），优先通过 Chrome DevTools MCP 读取；若没有浏览器/MCP 授权，请让用户提供可访问内容、截图、导出的 Markdown/PDF。

### MasterGo 原型处理

如果用户提供的prd里包含 MasterGo 链接（含 `mastergo.com`、`fileId`、`layerId`），**必须**使用 `mastergo-magic-mcp` 的 `getDsl` 工具读取设计稿 DSL 数据，提取页面宽度、布局方式、组件层级、设计 token、交互状态，映射到前端组件和布局描述中。

## 工作流程

### 步骤 0：选择生成模式

| 动作 | 说明 |
| ---- | ---- |
| 判断用户意图 | 只给 PRD/原型时进入 Outline Mode；明确要求完整可开发版时进入 Implementation Mode |
| 识别历史上下文 | 如果用户提供上一轮提纲，先核对是否有模块、页面、接口或权限变更 |
| 声明当前模式 | 输出前明确说明本轮是"提纲版"还是"AI 可落地开发版" |
| 保护已确认结论 | 进入 Implementation Mode 后必须继承已确认提纲，不得无理由重新拆模块 |

### 步骤 1：确认输入范围

| 动作 | 说明 |
| ---- | ---- |
| 识别输入 | 识别用户给出的 PRD、原型、UI、旧设计书、模块划分 |
| 处理不可访问链接 | 说明限制并请求粘贴关键内容或提供本地文件 |
| 复用已有初稿 | 如果已有设计书初版，先复用其模块和业务信息，再按本 skill 结构重排补齐 |
| 核对 PRD 需求 | **必须**逐一核对 PRD 需求列表，确保所有需求模块都被覆盖 |
| 禁止编造 | 不编造不存在的文件路径、组件名、权限码；不确定内容标 `(待确认)` |

### 步骤 1.5：Outline Mode 输出提纲版

当处于 Outline Mode 时，只输出提纲确认版。提纲版可以直接在对话中输出，也可以写入 `00_提纲确认.md`。

提纲版必须包含：
- 需求摘要：业务目标、范围、输入来源、缺失材料。
- 模块拆分表：模块、页面、核心能力、优先级、依赖、风险。
- 页面清单：页面/弹窗/抽屉、入口、主要交互、复用方向。
- 关键流程：主链路步骤或 ASCII 流程。
- 字段/API/权限初步假设：只列候选和确认点，不写最终契约。
- 二阶段补齐清单：说明 Implementation Mode 将补齐 Evidence、13 章节、代码块、字段字典、API 契约、状态机、验收标准。
- 待确认问题：以结构化表格列出问题、影响、推荐默认值、确认后修改范围。
- 提纲自检：检查 PRD 模块覆盖、页面覆盖、关键风险、是否可进入第二阶段。

提纲版输出完成后必须停在 `confirm` 状态，等待用户确认或补充；不要自动生成完整落地版，除非用户明确要求继续。

### 步骤 2：读取证据并提取事实

**当项目目录是代码仓库且处于 Implementation Mode 时，本步骤为强制执行。Outline Mode 只做必要探索。**

必须优先使用 Codegraph MCP；Codegraph 不能覆盖的内容，再使用文件读取工具扫描以下内容：

| 探索项           | 必须读取的内容                            | 产出                                    |
| ---------------- | ----------------------------------------- | --------------------------------------- |
| 技术栈           | `package.json`、`tsconfig.json`、构建配置 | 框架版本、UI 库、表格库、状态管理       |
| 目录结构         | `src/` 或 `pages/` 目录树                 | 页面组织方式、公共组件位置              |
| 列表页模式       | 找到至少 2 个现有列表页的实现文件         | `createPage` / 自定义列表模式、分页字段 |
| 权限模式         | `hasAuth` / `useAuth` / `codes` 定义方式  | 权限码格式、按钮控制方式                |
| 请求封装         | `request/` 或 `api/` 目录                 | APIMap 格式、接口注册方式               |
| 表单模式         | `useFormItem` / 表单项配置方式            | 控件类型、字典消费方式                  |
| 相关模块         | 当前需求最相关的页面文件                  | 挂载点、已有组件可复用项                |
| 上传/导入        | 现有上传组件、Excel 处理工具              | 可复用的上传/导入能力                   |
| 路由配置         | 路由注册文件                              | 路由定义格式、懒加载方式                |
| 类型定义         | 现有 typing.ts / types.ts                 | 类型命名约定                            |
| Store/Composable | stores/ 或 composables/ 目录              | Store 定义方式、命名约定                |
| 样式方案         | CSS/Less/UnoCSS 配置文件                  | 样式文件组织方式、设计 token            |

**Implementation Mode 强制输出**：每个探索项**必须**在设计书中以"代码库证据清单"（6 列表格）输出，并以 `Evidence(path:line)` 格式引用代码位置。

**Evidence 最低要求**：代码引用 ≥ 15 处 + PRD 引用 ≥ 8 处 = 总计 ≥ 23 处。

## TODO 管理规则

所有 TODO **必须**使用 9 列结构化格式（见 ⛔ 硬约束 B）。

| 阻塞级别        | 处理方式                                                                  |
| --------------- | ------------------------------------------------------------------------- |
| **是（P0 阻塞）** | 必须在临时方案中写清 mock contract 和后续替换点；必须标注影响哪个实现步骤 |
| **否（P1 待定）** | 可基于命名约定推导暂定值，标 `(待确认)`；标注确认后需要修改哪些章节       |
| **否（P2 优化）** | 记录即可，不影响主流程实现                                                |

约束：
- `TODO` 不能替代整个章节内容
- P0 阻塞型 TODO 必须明确阻塞哪个实现步骤
- `暂定` 只能用于结构清晰但名称/字段/路径未最终确认的内容
- 如果可以 mock，必须写清 mock contract 和后续替换点

## 编写规范与输出风格

- 使用中文输出，保持可直接复制进设计书的 Markdown 格式
- **表格优先于长文本**，列表优先于大段叙述
- **Evidence 引用**：引用代码库事实时**必须**带 `Evidence(path:line)` 格式；引用 PRD 事实时标注 `PRD:行号` 或章节名
- 字段、接口、权限码不确定时不编造，统一标 `(待确认)`
- 路由详情页与编辑页**优先**复用同一路由，通过 `?id=xxx&mode=edit` 或 `?mode=view` 区分
- 列表页**必须**沿用项目现有列表构建方式（如 `createPage`），并附 Evidence
- 权限控制**必须**沿用项目现有方式（如 `useAuth` / `hasAuth`）
- 公共组件集中放在 `00_总览与公共约定`，模块章节只引用
- 原型中的视觉信息要落到前端可执行描述：组件、布局、交互、状态、校验
- 对 PRD 与原型冲突的内容，单独列"冲突/待确认事项"
- 代码块必须标注语言类型（typescript / text / markdown）
- 对长文档默认先输出 Outline Mode 提纲；用户确认后再进入 Implementation Mode 补齐完整目录和模块设计书
- 如果用户明确要求一次性完整输出，则完整生成，不省略章节
