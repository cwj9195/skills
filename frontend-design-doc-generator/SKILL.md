# Frontend Design Doc Generator v2.0.0

根据 PRD、原型/UI 设计、现有代码库和开发者补充信息，生成可直接指导开发的前端项目设计书。

## 版本标识（重要！）

本 skill 当前版本为 **v2.0.0**。生成的每份设计书**必须**在基础信息表格中包含以下追溯字段：

| 追溯字段 | 值 | 说明 |
| --- | --- | --- |
| 生成工具 | frontend-design-doc-generator | 固定值 |
| 工具版本 | v2.0.0 | 取本 skill 的 version |
| 生成时间 | 当前日期 | 设计书首次生成时间 |

如果输出中缺少这三个字段，说明**没有使用本版本的 skill**。

## 何时使用

- 用户提供 PRD、CoDesign/蓝湖/Figma/MasterGo/截图等原型或 UI 信息，需要输出前端设计书
- 用户要求把需求拆成前端模块、页面、路由、组件、接口和状态流转
- 用户已有设计书初稿，需要按统一结构补全或重写
- 用户要求对照现有项目技术栈、组件库、路由和 Store 模式生成开发方案

## 输入材料

优先收集以下信息，不完整时也可以先产出草稿，并把缺失项标为 `TODO`：

| 材料 | 用途 | 缺失时处理 |
| --- | --- | --- |
| PRD 需求文档 | 业务逻辑、字段定义、权限矩阵、生命周期规则、审批规则 | 标注 `TODO：待 PRD 确认` |
| 原型/UI 设计 | 页面布局、交互细节、组件样式、状态流转 | 标注 `TODO：待原型确认` |
| 现有代码库 | 技术栈、组件库、路由约定、Store 模式、接口封装方式 | 读取代码后对齐现有实现 |
| 模块划分 | 章节顺序、优先级、开发边界 | 先基于 PRD 自动拆分，再请用户确认 |
| 开发者补充 | 特殊约束、字段命名、接口约定、兼容逻辑 | 作为最高优先级输入 |

如果链接需要登录（如语雀、CoDesign、Confluence、私有 GitHub），不要使用 WebFetch 直接抓取；请让用户提供可访问内容、截图、导出的 Markdown/PDF，或在有浏览器/MCP 授权时通过对应工具读取。

### MasterGo 原型处理

如果用户提供 MasterGo 链接（含 `fileId`、`layerId`），**必须**使用 `mastergo-magic-mcp` 的 `getDsl` 工具读取设计稿 DSL 数据，从中提取：

- 页面宽度、布局方式（flex/grid/absolute）
- 组件层级关系
- 字体、颜色、间距等设计 token
- 交互状态（hover/active/disabled）

将 DSL 信息映射到前端组件和布局描述中，而不是仅写"按原型"。

## 工作流程

### 步骤 1：确认输入范围

1. 识别用户给出的 PRD、原型、UI、旧设计书、模块划分。
2. 如果用户只给链接且无法访问，说明限制并请求粘贴关键内容或提供本地文件。
3. 如果已有设计书初版，先复用其模块和业务信息，再按本 skill 的结构重排补齐。
4. 不要臆造不存在的接口路径、字段 key、权限码；不确定内容统一标 `暂定` 或 `TODO`。

### 步骤 2：代码库探索（强制执行）

**当项目目录是代码仓库时，本步骤为强制执行，不可跳过。**

必须使用 Agent（Explore 类型）或文件读取工具，扫描以下内容并记录发现：

| 探索项 | 必须读取的内容 | 产出 |
| --- | --- | --- |
| 技术栈 | `package.json`、`tsconfig.json`、构建配置 | 框架版本、UI 库、表格库、状态管理 |
| 目录结构 | `src/` 或 `pages/` 目录树 | 页面组织方式、公共组件位置 |
| 列表页模式 | 找到至少 2 个现有列表页的实现文件 | `createPage` / 自定义列表模式、分页字段 |
| 权限模式 | `hasAuth` / `useAuth` / `codes` 定义方式 | 权限码格式、按钮控制方式 |
| 请求封装 | `request/` 或 `api/` 目录 | APIMap 格式、接口注册方式 |
| 表单模式 | `useFormItem` / 表单项配置方式 | 控件类型、字典消费方式 |
| 相关模块 | 当前需求最相关的页面文件 | 挂载点、已有组件可复用项 |
| 上传/导入 | 现有上传组件、Excel 处理工具 | 可复用的上传/导入能力 |

每个探索项必须在设计书中以 `Evidence(path:line)` 格式引用代码位置，格式为：

```
Evidence(`apps/pim/pages/preselected/index.tsx:216`)
```

### 步骤 3：模块拆分

按业务闭环拆模块，而不是按接口或组件拆模块。每个模块至少明确：

| 维度 | 要回答的问题 |
| --- | --- |
| 页面 | 有哪些列表、详情、新建、编辑、审批、弹窗、抽屉 |
| 数据 | 页面初始化、查询、提交、审批、导出、上传的链路 |
| 状态 | 草稿、申请中、已通过、已驳回、已撤销、待执行等状态如何影响 UI |
| 权限 | 按钮级、行级、数据级分别怎么控制 |
| 复用 | 复用哪些现有组件，哪些需要新建公共组件 |
| 风险 | 哪些字段、接口、规则需要后端/产品确认 |

### 步骤 4：按固定章节输出

每个业务模块必须包含以下 **7 个**子章节（v2 新增状态机、字段字典、文件变更计划），格式参考 [输出模板](./references/output-template.md)：

1. **页面结构** — 路由、层级、布局、交互
2. **数据链路** — 初始化、查询、提交、联动
3. **状态机与交互规则** ⭐ v2 新增 — 状态枚举、状态转换矩阵、各状态下 UI/按钮可用性
4. **字段字典** ⭐ v2 新增 — 按 PRD 逐字段列出 key、控件类型、必填、校验规则
5. **组件复用/沿用** — 复用组件 + 新建组件 + 通信方式
6. **相关接口** — 接口表格（方法、路径、入参、出参、调用时机）
7. **补充逻辑** — 边界条件、权限控制、校验规则、性能考量

公共组件、公共权限、公共路由规范、公共数据字典放在"公共约定"章节，模块内只引用，不重复长篇定义。

### 步骤 5：文件变更计划（v2 新增，强制执行）

必须在设计书末尾（测试要点之前）输出文件变更计划表：

| 类型 | 操作 | 路径 |
| --- | --- | --- |
| 页面 | 修改 | 具体文件路径 — 修改内容简述 |
| 组件 | 新建 | 具体文件路径 |
| API | 修改 | 具体文件路径 — 新增接口 |

路径必须基于步骤 2 的代码库探索结果，不允许臆造路径。

### 步骤 6：自检（强制执行）

输出完成后，**必须**在末尾追加自检结果，格式如下：

```markdown
## 自检结果

| 检查项 | 状态 | 说明 |
| --- | --- | --- |
| 基础信息含工具版本 | ✅/❌ | |
| 每个模块含 7 个子章节 | ✅/❌ | |
| 状态机已定义 | ✅/❌ | |
| 字段字典已列出 | ✅/❌ | |
| 文件变更计划已列出 | ✅/❌ | |
| Evidence 引用 ≥ 5 处 | ✅/❌ | 共 N 处 |
| TODO/待确认事项已列出 | ✅/❌ | |
| 实现步骤已列出 | ✅/❌ | |
```

## 输出要求

### 文档结构

推荐顶层结构：

1. 基础信息（**必须包含工具版本追溯字段**）
2. 版本履历
3. 需求概述
4. 前端模块总览
5. 公共约定
6. 业务模块设计（逐模块，每模块 7 个子章节）
7. 文件变更计划 ⭐ v2 新增
8. 实现步骤 ⭐ v2 新增
9. 路由汇总
10. 接口清单
11. TODO / 待确认事项
12. 测试要点
13. 自检结果 ⭐ v2 新增

### 每个模块必须覆盖

#### 1. 页面结构

必须包含：

- 路由定义：路径、query 参数、同一路由复用情况
- 页面层级关系：列表 → 详情 → 弹窗/抽屉/审批区，用树形结构表示
- 布局逻辑：用 ASCII 图表示筛选区、操作栏、表格区、分页栏、详情区
- 元素排布：筛选项、操作按钮、表格列、底部操作区的位置关系
- 交互细节表：场景/触发方式/前端行为/反馈（表格形式）

#### 2. 数据链路

必须包含：

- 页面初始化加载哪些接口（伪代码流程图 + 表格）
- 查询/筛选/分页/排序的请求时机和参数组装
- 新建/编辑/审批/撤销/导入/导出的请求链路（每个链路一段伪代码流程）
- Store 缓存字段、是否持久化、何时清空
- 数据变更后的联动刷新逻辑

#### 3. 状态机与交互规则 ⭐ v2 新增

**每个有状态流转的模块必须包含状态机定义。**

必须包含：

- 状态枚举表：状态值 / 中文名 / UI 表现 / 允许动作 / 禁止动作
- 状态转换图（伪代码或文本形式）：`状态A → 触发事件 → 状态B`
- 行级状态矩阵：当不同行可以有不同状态时（如导入表），用矩阵表列出各整体状态 × 行状态下的编辑/删除/操作可用性

示例格式：

```markdown
| 状态 | UI 表现 | 允许动作 | 禁止动作 |
| --- | --- | --- | --- |
| idle | 上传区域展示拖拽/点击 | 选择文件 | — |
| uploading | 文件名 + 进度条 + 取消 | 取消上传 | 再次上传 |
| error | 失败原因 message | 重新上传 | — |
| uploaded | 成功 message | 查看记录 | 重复提交 |
```

#### 4. 字段字典 ⭐ v2 新增

**每个涉及表单/表格的模块必须列出完整字段字典。**

按分组列出所有字段，每个字段必须包含：

| 字段名 | key | 控件类型 | 必填 | 校验规则 | 枚举值/数据源 |

如果字段数量多（>20 个），按业务分组列出（如 SPU 信息、价格信息、规格信息）。

字段定义必须与 PRD 中的字段表逐一对应，不允许遗漏。PRD 中每个字段的"填写规则"和"特殊说明"必须映射到校验规则列。

#### 5. 组件复用/沿用

必须包含：

- 复用现有组件，并标注来源 + Evidence：组件库、业务公共组件、当前模块组件
- 需要新建的组件：中文组件名、职责、props、emit、使用场景
- 组件间通信矩阵：来源组件 → 目标组件 → 通信方式 → 数据

组件名称使用中文，如 `<商品选择器 />`，不要只写英文组件名。

#### 6. 相关接口

必须包含接口表格：

| 接口名称 | 方法 | 路径 | 入参 | 出参 | 调用时机 | 备注 |
| --- | --- | --- | --- | --- | --- | --- |

如果接口未定，路径写 `TODO：待后端确认`；字段 key 未定，标注 `暂定`。

#### 7. 补充逻辑

必须包含：

- 边界条件：空数据、超长文本、重复提交、并发操作、接口失败
- 权限控制：按钮级、行级、数据级
- 校验规则：前端表单校验（前端做/不做 + 原因）、业务规则校验（后端为主 + 前端展示）
- 性能考量：虚拟滚动、防抖节流、懒加载、批量操作分片
- 审批/生命周期/状态机等额外业务规则

## 编写规范

- 表格优先于长文本，列表优先于大段叙述。
- **Evidence 引用**：引用代码库事实时必须带 `Evidence(path:line)` 格式；引用 PRD 事实时标注 `PRD:行号` 或章节名。
- 字段、接口、权限码不确定时不要编造，统一标 `暂定` 或 `TODO`。
- 路由详情页与编辑页优先复用同一路由，通过 `?id=xxx&mode=edit` 或 `?mode=view` 区分，除非项目已有不同约定。
- 列表页优先沿用项目现有列表构建方式；如果项目已有 `createPage`，明确写"使用 createPage 构建"并附 Evidence。
- 权限控制优先沿用项目现有方式；如果项目已有 `useAuth` / `hasAuth`，在按钮级、行级、数据级分别说明。
- 公共组件集中放在"公共约定"，模块章节只引用。
- 原型中的视觉信息要落到前端可执行描述：组件、布局、交互、状态、校验，不只描述"按原型"。
- 对 PRD 与原型冲突的内容，单独列"冲突/待确认事项"。

## 质量检查清单

输出前检查：

- [ ] 基础信息表是否包含 `生成工具`、`工具版本`、`生成时间` 三个追溯字段
- [ ] 每个模块是否都有 7 个固定子章节（页面结构/数据链路/状态机/字段字典/组件/接口/补充逻辑）
- [ ] 每个列表页是否有筛选区、操作区、表格列、分页/排序说明
- [ ] 每个详情/编辑/新建页是否有路由、表单字段、校验、保存/提交链路
- [ ] 每个有状态流转的模块是否有状态枚举表 + 状态转换规则
- [ ] 每个表单/表格模块是否有完整字段字典（与 PRD 字段一一对应）
- [ ] 每个导入/导出/上传是否有文件格式、触发时机、错误反馈
- [ ] 是否列出文件变更计划（具体文件路径 + 操作类型）
- [ ] 是否列出实现步骤（有序列表）
- [ ] 是否列出接口、字段、权限、原型缺失的 TODO
- [ ] 是否对齐现有技术栈和组件库，而不是输出通用方案
- [ ] Evidence 引用是否 ≥ 5 处（代码库引用 + PRD 引用）
- [ ] 自检结果是否已追加到文档末尾

## 输出风格

- 使用中文输出。
- 保持可直接复制进设计书的 Markdown 格式。
- 对长文档先输出完整目录和一个模块样例；用户确认后再继续补齐剩余模块。
- 如果用户明确要求一次性完整输出，则完整生成，不省略章节。
- 字段字典、状态机等密集信息**必须用表格**，不要用列表或描述性段落。
