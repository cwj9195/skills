# Front Design Agent v2 工作流

## 0. Workflow-first 定位

本状态机是 **Workflow-first, Agent-inside** 的执行约束，不是纯 Agent 自由规划。

- 外层 Workflow 固定阶段、输入、产物和停止条件。
- 内层 Agent 负责信息提取、冲突判断、代码模式归纳、接口推导、任务拆解和代码生成。
- `mode_select` 是 workflow 路由，不是新增业务模式。
- `module-split`、`module-design`、`implement` 是 workflow 主阶段。
- `clarify`、`evidence`、`code_ref`、`api_contract`、`analyze`、`verify` 是阶段内子流程。
- 能由规则确定的，不交给 Agent 自由决策；证据不足、路径冲突或接口状态不满足时进入 `repair / clarify`。
- 不新增第四种模式，不新增默认文件，不引入独立 workflow runtime。

## 1. 状态机（v2.3 模式交互版）

| 状态                  | 进入条件                                          | 行为                                                                           | 下一状态                                        |
| --------------------- | ------------------------------------------------- | ------------------------------------------------------------------------------ | ----------------------------------------------- |
| `initial`             | 收到 PRD、原型、MasterGo、模块目录或任务包需求    | 识别输入材料                                                                   | `mode_select`                                   |
| **`mode_select`**     | 已识别输入材料                                    | 自动判断 `module-split` / `module-design` / `implement`；无法判断时询问用户    | `module_split` / `clarify` / `implement`        |
| **`module_split`**    | 用户提供 PRD/UI/原型，且未指定模块目录或模块文件  | 拆分模块，生成 Manifest `模块索引.md` 与模块文件夹，等待人工审查/补图               | `done` / `repair`                               |
| **`clarify`**         | 进入 `module-design` 且模块材料存在               | **苏格拉底式澄清：分段提问，消除歧义**                                         | `evidence`                                      |
| `evidence`            | 澄清完成，用户要求单模块 AI 落地版                | 使用 MCP 采集 Evidence                                                         | `code_ref`                                      |
| `code_ref`            | Evidence 达标                                     | **CodeReferenceAgent：提取项目级代码模式**                                     | `api_contract` / `repair`                       |
| **`api_contract`**    | `module-design` 必经                              | 生成或更新根级 `接口契约.md`，维护接口 ID、状态和请求/响应契约；无接口模块写入“模块接口覆盖记录” | `design` / `repair`                             |
| `design`              | 代码参考.md 已生成                          | 生成模块实现方案（引用代码参考，分段呈现）                              | **`analyze`**                                   |
| **`analyze`**（新增） | 前端设计.md 已生成                         | **强制自审：模拟实现测试 + 交叉引用检查**                                      | `task` / `design` / `repair`                    |
| `task`                | 自审通过                                          | 生成代码级完成标准的任务包（粒度 2-5 分钟）                                    | `verify`                                        |
| **`verify`**（新增）  | 编码任务.md 已生成                        | **验证：任务粒度检查 + 完成标准可断言检查**                                    | `done` / `task`                                 |
| **`implement`**       | 用户提供 `编码任务.md` 或要求按任务写代码 | 只按任务执行代码改动，输出改动文件、验证结果、Regression Check、Defensive Code | `done` / `repair`                               |
| `repair`              | Evidence 不足 / 自审不通过 / 验证失败 / 任务阻塞  | 列阻塞项和临时方案                                                             | `mode_select` / `clarify` / `evidence` / `task` |
| `done`                | 当前模式完成                                      | 交付当前模式产物                                                               | —                                               |

### 状态转换图

```text
initial → mode_select ┬→ module_split → done
                      ├→ clarify → evidence → code_ref → api_contract → design → analyze → task → verify → done
                      └→ implement → done
                                  ↑        ↑          ↑               ↑        ↑        ↑
                                  └──────── repair ←──┴───────────────┴────────┴────────┘
```

### 关键变更（v1 → v2 → v2.3）

| v1       | v2/v2.1                                | v2.3                                   | 原因                                           |
| -------- | -------------------------------------- | -------------------------------------- | ---------------------------------------------- |
| 无澄清   | **clarify 阶段**                       | 保留 clarify，但只用于 `module-design` | 拆模块阶段先做事实切片，设计阶段再消除实现歧义 |
| 直接生成 | **analyze 自审**                       | 保留 analyze                           | 必须验证"只看 design 能否写代码"（Spec Kit）   |
| 无验证   | **verify 阶段**                        | 保留 verify                            | 任务粒度 + 完成标准可断言（Superpowers）       |
| 单一路径 | evidence → code_ref → design → analyze | **mode_select + 三模式 + 默认 API 契约** | 用户不需要写长提示词，skill 主动选择阶段       |

## 2. 模式选择

收到请求后先判断模式，并在回复开头声明当前模式、本轮产出和本轮不会做什么。

### 中文命名与旧名兼容

新生成的产物必须使用中文名；读取历史产物时，如果中文名不存在，按以下英文旧名兼容读取。Manifest 中记录中文路径；如果本轮读取到英文旧名，后续更新 Manifest 时迁移为中文路径。

| 中文默认名 | 英文兼容名 |
| ---------- | ---------- |
| `模块索引.md` | `module-index.md` |
| `需求.md` | `requirement.md` |
| `界面.md` | `ui.md` |
| `原型.md` | `prototype.md` |
| `审查记录.md` | `review-notes.md` |
| `设计/` | `design/` |
| `需求说明.md` | `requirements.md` |
| `代码参考.md` | `code-reference.md` |
| `接口契约.md` | `api-contract.md` |
| `前端设计.md` | `frontend-design.md` |
| `编码任务.md` | `claude-code-task.md` |

稳定 ID 不中文化：`M{N}`、`API-Mx-xxx`、`T-Mx-xx` 必须保持机器可读。

| 模式            | 自动进入条件                                               | 输出                                                                                      | 禁止                                       |
| --------------- | ---------------------------------------------------------- | ----------------------------------------------------------------------------------------- | ------------------------------------------ |
| `module-split`  | 用户提供 PRD/UI/原型，且未指定模块目录或模块文件           | `模块索引.md`、各模块 `需求.md` / `界面.md` / `原型.md` / `审查记录.md` | 不生成 `接口契约.md`、设计书、任务包或代码 |
| `module-design` | 用户指定模块目录、模块文件，或明确要求针对某模块生成设计书 | `需求说明.md`、`代码参考.md`、`接口契约.md`、`前端设计.md`、`编码任务.md` | 不把完整 PRD 作为主要输入；不处理无关模块  |
| `implement`     | 用户提供 `编码任务.md` 或要求按任务写代码          | 代码改动、验证结果、Regression Check、Defensive Code                                      | 不生成设计书；不回读完整 PRD；不改无关模块 |

无法判断时，先问用户选择：`module-split`、`module-design` 或 `implement`。

进入模式时使用：

```text
我将使用 {mode} mode。
本轮会产出：{outputs}。
本轮不会做：{non_goals}。
```

## 3. `module-split` 模块拆分阶段

### 触发条件

用户提供 PRD、MasterGo UI、原型或截图，并希望先按模块拆分供人工审查。

用户可以只有 PRD；若 PRD 模块内含 MasterGo 图层链接，先作为 UI 线索记录，必须尝试 `mastergo-magic-mcp.getDsl`。只有 DSL 调用成功后，才能作为 Verified UI Evidence 写入对应模块的 `界面.md`；调用失败时写入 `审查记录.md`，标记 Fallback / Unverified。没有 UI/原型证据时，模块仍可拆分，但必须在 `界面.md` / `原型.md` / `审查记录.md` 标注暂无证据和待补充项。

### 输出结构

```text
模块索引.md
M1-{中文模块名}/
  需求.md
  界面.md
  原型.md
  审查记录.md
M2-{中文模块名}/
  需求.md
  界面.md
  原型.md
  审查记录.md
```

### 文件职责

| 文件              | 内容                                                                                        |
| ----------------- | ------------------------------------------------------------------------------------------- |
| `模块索引.md` | Manifest 单一入口源：设计根、模块 ID、路径、状态、接口 ID 索引、人工审查状态和跨模块规则 |
| `需求.md`  | 模块需求、范围、字段、权限、接口线索、验收标准                                              |
| `界面.md`           | UI 页面、组件结构、视觉/交互要点、截图或图层引用                                            |
| `原型.md`    | 原型流程、页面跳转、状态流转                                                                |
| `审查记录.md` | 人工审查点、PRD/UI/原型冲突、待确认问题                                                     |

### 约束

- 只做模块事实切片，不生成 `代码参考.md`、`前端设计.md`、`编码任务.md`。
- `模块索引.md` 是 Manifest，不是事实大杂烩；只维护模块身份、路径、状态和索引。
- Manifest 必须包含设计根目录、根级 `接口契约.md` 路径、模块 ID、模块目录、输入文件路径、设计输出目录、模块状态、人工审查状态、接口 ID 索引和 Evidence。
- Manifest 禁止维护完整需求、完整 UI DSL、完整请求响应结构、代码骨架和任务明细。
- `M{N}` 是稳定主键，目录名可调整；后续模块定位以 Manifest 的模块 ID 为准。
- 不生成正式 `接口契约.md`；只在 `需求.md` 记录接口线索，例如列表查询、导入、导出、校验、提交。
- 不写代码。
- 所有事实必须带 Evidence。
- PRD、UI、原型冲突时写入 `审查记录.md`，不得擅自裁决为事实。
- 不确定接口写入 `审查记录.md` 作为人工审查项，不得提前确定 method/path/字段为已确认事实。
- `module_split` 完成后停在人工审查/补图阶段，不自动进入 `module-design`。

## 4. `api_contract` 默认接口契约阶段

### 触发条件

进入 `module-design` 后必须执行，生成或更新设计根目录的 `接口契约.md`。

### 输出规则

- `接口契约.md` 是接口契约唯一事实源，包含接口 ID、模块、场景、method、path、请求参数、响应结构、错误码、权限、状态。
- 接口状态只能是 `Draft`、`Reviewed`、`Confirmed`、`Changed`。
- 当前模块的 `需求说明.md` 只保留接口需求摘要；`前端设计.md` 只引用 `接口契约.md#API-Mx-xxx` 并说明前端接入方式；`编码任务.md` 只引用接口 ID 和停止条件。
- 接口契约不绑定 dev/test/prod 环境；baseURL 与环境映射由项目请求层和部署环境控制。
- 当前模块无新增/变更接口时，也要在 `接口契约.md` 的“模块接口覆盖记录”中记录“无新增/变更接口”，不得跳过该文件；接口总览只放真实接口 ID。
- `接口契约.md` 固定位于设计根目录，例如 `apps/pim/docs/设计/接口契约.md`；单模块设计产物位于 `{module-dir}/设计/` 时，默认相对引用为 `../../接口契约.md`。
- 多模块并行设计时，只允许更新当前模块 section 和全局总览中当前模块对应行，不得重写其他模块接口。
- Draft 接口允许使用 `Proposed path`，但必须标记 `Draft` / `Inferred` / `(待确认)`；不得作为真实联调任务的非阻塞前提。

### 人工介入时机

- 第一轮后：人工审查模块拆分和接口线索。
- 第二轮后：人工审查 `接口契约.md` 中当前模块接口，并把状态推进到 `Reviewed` 或 `Confirmed`。
- 第三轮前：本轮任务依赖的接口状态至少为 `Reviewed`；涉及真实联调必须为 `Confirmed`。

## 5. `/clarify` 苏格拉底式澄清阶段（来自 Spec Kit）

### module-design 模块定位

如果用户只提供 `模块索引.md` 和模块编号/名称（如 M2），必须先读取 Manifest 中的 `模块索引.md#M{N}` 记录。该引用必须对应 `模块索引.md` 内的 `<a id="M{N}"></a>` 稳定锚点和 `### M{N}: 模块名` 标题；路径事实以 Manifest 表为准。读取后定位模块输入和输出，并自动读取：

- `{module-dir}/需求.md`
- `{module-dir}/界面.md`
- `{module-dir}/原型.md`
- `{module-dir}/审查记录.md`
- `{module-dir}/设计/` 输出目录
- 设计根目录下的 `接口契约.md`

只有 Manifest 缺少该模块、目录不存在、输入文件缺失或输出目录不可推导时，才向用户索要路径或补充材料。

### 触发条件

进入 `module-design` 后必须先执行澄清检查；`module-split` 只做模块事实切片，不强制逐模块澄清。

### 执行规则

0. **先检查再提问**：如果 `审查记录.md` 无阻塞问题、Manifest 人工审查状态为已审查，且需求/UI/原型证据足以进入设计，可在 需求说明.md 的「澄清记录」写明“无需补问”的依据后继续，不机械提问。
1. **分段提问**：不要一次性抛出 20 个问题。每轮最多 3-5 个问题，等用户回答后再问下一轮。
2. **聚焦关键歧义**：
   - 边界条件（数据为空、超长、并发、权限不足时怎么处理？）
   - 异常路径（网络超时、接口失败、表单校验失败时的 UI 表现？）
   - 权限差异（不同角色看到的内容差异？按钮显隐规则？）
   - 模块交互（跨模块的数据流？共享状态？路由跳转？）
3. **记录澄清结论**：每轮澄清结论写入 需求说明.md 的「澄清记录」段落。
4. **PRD 矛盾处理**：如果 PRD 本身有矛盾，记录到 TODO 并给出推荐方案，让用户选择。
5. **退出条件**：当前模块关键歧义已消除；若无需补问，必须写明依据；若需要补问，必须完成问题回答或把阻塞项写入 TODO。

### 澄清问题模板

```text
## 澄清轮次 {N}

### Q1: {模块名} - {边界/异常/权限/交互}
{具体问题描述}
- A) {选项A}
- B) {选项B}
- C) {选项C}

### Q2: ...
```

## 6. `/analyze` 强制自审阶段（来自 Spec Kit）

### 触发条件

前端设计.md 写完后，**必须执行自审**，不可跳过。

### 自审检查项（逐条执行）

| 序号 | 检查项           | 验证方法                                                            | 不通过动作         |
| ---- | ---------------- | ------------------------------------------------------------------- | ------------------ |
| 1    | **模拟实现**     | 只看 代码参考.md + 接口契约.md + 前端设计.md 的某个模块，尝试写出 index.tsx 骨架 | 回补缺失的代码骨架 |
| 2    | **代码粘贴测试** | 前端设计中"可直接粘贴"的代码块，检查语法和 import 路径               | 修正代码           |
| 3    | **完成标准测试** | 每个任务完成标准能否写成 assert 断言                                | 改写为代码级断言   |
| 4    | **交叉引用**     | 前端设计引用的 代码参考 章节号是否真实存在                     | 修正引用           |
| 5    | **缺失模式**     | 是否有模块需要 代码参考 未覆盖的代码模式                      | 回补新章节         |
| 6    | **粒度检查**     | 每个任务是否可在 2-5 分钟内完成                                     | 拆分过大的任务     |

### 自审结果输出格式

```text
## /analyze 自审结果

| 序号 | 检查项   | 结果  | 缺失内容 | 回补动作 |
| ---- | -------- | ----- | -------- | -------- |
| 1    | 模拟实现 | ✅ / ❌ | TODO     | TODO     |
| 2    | 代码粘贴 | ✅ / ❌ | TODO     | TODO     |
| 3    | 完成标准 | ✅ / ❌ | TODO     | TODO     |
| 4    | 交叉引用 | ✅ / ❌ | TODO     | TODO     |
| 5    | 缺失模式 | ✅ / ❌ | TODO     | TODO     |
| 6    | 粒度检查 | ✅ / ❌ | TODO     | TODO     |

通过：{N}/6
```

### 退出条件

- ✅ 6/6 通过 → 进入 `task` 阶段
- ⚠️ 4-5/6 通过 → 就地回补缺失内容，重新自审
- ❌ ≤ 3/6 通过 → 回到 `evidence` 阶段重新采集

## 7. `verify` 最终验证阶段（来自 Superpowers）

### 触发条件

编码任务.md 写完后执行。

### 验证检查项

| 检查项         | 标准                       | 验证方法                     |
| -------------- | -------------------------- | ---------------------------- |
| 任务粒度       | 每个任务 2-5 分钟可完成    | 检查任务涉及文件数和代码量   |
| 精确文件路径   | 每个任务列出精确的文件路径 | grep 检查 TODO 路径          |
| 验证步骤       | 每个任务有明确的验证步骤   | 检查每个任务是否有"验证"段落 |
| 完成标准可断言 | 完成标准能转成 assert      | 尝试把标准写成伪代码断言     |
| 依赖关系       | 任务间依赖正确标注         | 检查前置依赖是否已定义       |
| 回归覆盖       | 回归检查清单覆盖已有功能   | 对比现有模块列表             |
| 接口契约       | 涉及接口的任务引用接口 ID，并写明状态停止条件；无接口任务写“无” | 检查 `接口契约.md#API-Mx-xxx` / `无` |
| 接口状态       | Draft 不进入真实联调；Mock/本地骨架至少 Reviewed；真实联调必须 Confirmed | 检查 接口契约 状态 |
| 路径引用       | 模块设计目录默认用 `../../接口契约.md` 指向根级契约 | 检查相对路径 |
| Manifest       | 模块 ID、模块锚点、模块目录、输入文件、设计输出目录、根级 `接口契约.md` 路径可解析 | 检查 `模块索引.md#M{N}` |

## 8. 分段呈现规则（来自 Superpowers）

### 核心原则

不要一次性抛出 500 行的长文档。分段呈现，让用户逐段确认。

### 执行规则

| 文档                | 分段方式                            | 每段长度    |
| ------------------- | ----------------------------------- | ----------- |
| module-split        | 按模块分段，每段 1 个模块文件夹     | ≤ 120 行/段 |
| 需求说明.md     | 按模块分段，每段 1 个模块           | ≤ 100 行/段 |
| 代码参考.md   | 按代码模式分段，每段 1 个模式       | ≤ 80 行/段  |
| 接口契约.md     | 按模块接口分段，每段 1 个模块       | ≤ 120 行/段 |
| 前端设计.md  | 按模块分段，先完整模块再差异模块    | ≤ 150 行/段 |
| 编码任务.md | 按模块分段，每段 1 个模块的任务清单 | ≤ 100 行/段 |

### 分段呈现流程

```text
1. 呈现第 1 段 → 等用户确认 → 呈现第 2 段 → 等用户确认 → ...
2. 如果用户对某段有修改意见 → 就地修改 → 重新确认 → 继续下一段
3. 全部段落确认后 → 合并为完整文档
```

### 例外

- 用户明确说"一次性输出"时，可以不分段
- 提纲版可以一次性输出（因为它本身是概览）

## 9. 输入处理

| 输入          | 处理方式                                               | Evidence                   |
| ------------- | ------------------------------------------------------ | -------------------------- |
| PRD           | 抽取业务目标、用户故事、需求项、验收、边界             | `PRD(section:line)`        |
| 模块文件夹    | 作为 `module-design` 的主要输入，只处理当前模块        | `Module(path:section)`     |

| 代码库        | Codegraph MCP 读取结构、路由、接口、组件、**代码模式** | `Evidence(path:line)`      |

## 10. MCP 使用规则

### Codegraph

优先顺序：

1. `codegraph_status`
2. `codegraph_files`
3. `codegraph_context` — **提取代码模式的关键入口**
4. `codegraph_explore` — **深入读取具体实现**
5. `codegraph_impact`

CodeReferenceAgent 的 Codegraph 探索顺序：

```text
1. codegraph_context("项目中最完整的列表页实现") → 找到参考页
2. codegraph_explore(参考页的关键符号) → 读取完整实现
3. codegraph_context("表单字段工厂") → 提取 formItem 工厂
4. codegraph_context("请求层 APIMap") → 提取请求模式
5. codegraph_context("路由注册") → 提取路由模式
6. codegraph_context("权限控制") → 提取权限模式
...（按需扩展）
```

若 Codegraph 未初始化：

```text
记录阻塞项：Codegraph 未初始化。
建议用户执行：codegraph init。
如必须继续，降级为只读 RTK 命令，可信度标记 Fallback。
```

### Chrome DevTools

用于访问：

- PRD 链接
- 原型链接
- 内部组件库文档
- 登录态系统页面

### MasterGo

检测到 MasterGo 链接时必须：

1. 解析 `fileId`、`layerId`。
2. 调用 `mastergo-magic-mcp.getDsl`。
3. 抽取页面宽度、栅格、布局、组件层级、token、交互状态。
4. 映射到公司组件库或业务公共组件。
5. 无法识别的图层进入 TODO。

不得只记录 MasterGo 链接而不尝试调用 `mastergo-magic-mcp.getDsl`。

- 调用成功：DSL 结构进入 `界面.md`；`module-design` 阶段作为 `Verified` Evidence 注入 `前端设计.md` 的 UI/布局章节。
- 调用失败：记录失败原因、降级方式、待补材料和证据来源；在 `module-split` 阶段写入 `审查记录.md`，在 `module-design` 阶段写入 TODO 或设计风险。
- 降级证据只能标记为 `Fallback` 或 `Unverified`，不得替代 MasterGo DSL 写成 `Verified`。

**DSL 结果处理规则**

DSL 返回的 JSON 可能很大（数百 KB），处理策略：

1. 如果 DSL 数据 ≤ 50KB，直接内联到 前端设计.md
2. 如果 DSL 数据 > 50KB，提取关键结构（组件树、尺寸、布局），生成精简版
3. 提取时关注：容器布局（flex/grid）、组件类型和嵌套层级、间距和尺寸 token
4. 忽略：纯装饰节点、重复的原子元素

## 11. Evidence 可信度

| 可信度              | 定义                                                     |
| ------------------- | -------------------------------------------------------- |
| `Verified`          | 来自 Codegraph、MasterGo DSL、API 文档或明确 PRD 行      |
| `Inferred`          | 基于现有命名约定、相邻模块实现推导                       |
| `Fallback`          | Codegraph 或链接不可用时，用只读命令、截图或用户描述补充 |
| `Unverified`        | 无可验证来源，必须进入 TODO                              |
| `Clarified`（新增） | 来自用户在 clarify 阶段的明确回答                        |

## 12. 输出顺序（v2.3）

### module-split 输出

| 5    | `M{N}-{中文模块名}/审查记录.md` | 冲突、待确认和人工审查点                  |

### module-design 输出

| 顺序 | 文档                  | 说明                                        |
| ---- | --------------------- | ------------------------------------------- |
| 1    | `需求说明.md`     | 需求事实 + 澄清记录                         |
| 2    | `代码参考.md`   | 代码参考手册                                |

## 13. 自检门槛（v2.3）

### module-split 自检

- [ ] `模块索引.md` 是 Manifest，只维护模块身份、路径、状态和索引，不复制详细事实。
- [ ] Manifest 包含设计根目录、根级 `接口契约.md` 路径、模块 ID、模块目录、输入文件、设计输出目录、模块状态、接口 ID 索引和人工审查状态。
- [ ] Manifest 中的 `M{N}` 是稳定主键，模块目录可随人工审查调整。
- [ ] 多模块并行时只更新当前模块 Manifest 行和当前模块 section。
- [ ] 每个模块目录都有 `需求.md`、`界面.md`、`原型.md`、`审查记录.md`。
