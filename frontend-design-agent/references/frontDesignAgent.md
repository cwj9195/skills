# FrontDesignAgent v2.3：前端设计书生成系统

## 1. 问题分析

### v1 的核心问题

| 问题               | 影响                                                  | v2 解决方式                   |
| ------------------ | ----------------------------------------------------- | ----------------------------- |
| 7 份文档冗余严重   | spec↔需求说明 重叠 70%，fact-set↔ci-gate 重叠 80% | **精简为 5 份核心文档**       |
| 缺少代码级模板     | API 签名和用法只写名字不写签名                        | **新增 代码参考.md**    |
| 设计数据是推断的   | MasterGo DSL 未实际调用                               | **强制 DSL 采集**             |
| 完成标准太模糊     | "列表加载+筛选+分页"无法指导 AI                       | **代码级完成标准**            |
| 13 章固定章节过重  | 小模块也写 13 章，填充废话                            | **完整版 vs 差异版**          |
| **无澄清阶段**     | 歧义未消除就开始写文档                                | **`/clarify` 苏格拉底式澄清** |
| **无自审验证**     | 前端设计写完不知道能不能用                             | **`/analyze` 模拟实现测试**   |
| **任务粒度无控制** | 一个大任务 AI 写半天                                  | **2-5 分钟粒度 + `/verify`**  |

### v2.3 设计原则

- **Workflow-first, Agent-inside**：工作流优先，智能体内嵌；外层固定三轮阶段，内层由 Agent 完成分析、归纳、判断和生成
- **设计书不是文档，是代码的蓝图**：AI 拿到后能直接写出可运行代码
- **先选模式再执行**：`module-split` 拆模块，`module-design` 做单模块设计，`implement` 按任务写代码
- **实现阶段约束最强**：`implement` 只能按 `编码任务.md` 的任务 ID 执行，不允许绕过任务包自由扩范围
- **Manifest 单一入口源**：`模块索引.md` 只维护模块身份、路径、状态和索引，不承载详细业务事实
- **中文产物命名**：新生成业务产物默认中文名；用户显式提供旧英文文件路径时可作为普通输入读取，但不作为默认兼容规则
- **代码参考.md 是基石**：没有代码参考手册，design 就是空中楼阁
- **5 份文档，各有分工**：需求说明（做什么）→ 代码参考（怎么写）→ 接口契约（接口契约）→ 前端设计（具体方案）→ 编码任务（做哪些）
- **先提取再设计**：CodeReferenceAgent 先于 DesignDocAgent 运行
- **澄清先于单模块设计**：`module-design` 必须 clarify；`module-split` 只做事实切片并记录待确认
- **证据优于声明**：所有"已完成"必须有可验证证据（来自 Superpowers）
- **分段呈现**：不要一次性抛出长文档，逐段确认（来自 Superpowers）

### 方法论来源

| 来源                    | 核心理念                                | 引入的能力                                                               |
| ----------------------- | --------------------------------------- | ------------------------------------------------------------------------ |
| **Spec Kit**（GitHub）  | 规格驱动开发，spec 是活的可执行产物     | `/clarify`（苏格拉底式澄清）、`/analyze`（模拟实现测试）、多步骤迭代精炼 |
| **Superpowers**（obra） | 完整开发方法论，TDD、系统化、降低复杂度 | 任务粒度控制（2-5 分钟）、分段呈现、证据优于声明、两阶段审查             |

## 2. 系统架构

### 2.1 总体架构（v2.3 三模式版）

```text
输入材料
  ├─ PRD / MasterGo / 原型 / 截图
  ├─ 模块文件夹
  └─ 编码任务.md
        ↓
mode_select
        ├─ module-split：PRD/UI/原型 → 模块索引.md + 模块文件夹
        ├─ module-design：单模块文件夹 + 代码库 → 5 份核心设计产物
        └─ implement：编码任务.md → 代码改动 + 验证结果
```

### 2.2 标准产物与链路入口（v2 精简版）

`模块索引.md` 是 `module-split` 产出的前置 Manifest，不计入 `module-design` 默认 5 份核心设计产物。

| 产物                  | 职责                                                     | 给谁                        |
| --------------------- | -------------------------------------------------------- | --------------------------- |
| `模块索引.md`     | Manifest 单一入口源：模块 ID、路径、状态、接口 ID 索引和审查状态 | 三轮链路入口                |
| `需求说明.md`     | 需求事实：模块、字段、权限、验收标准                     | DesignDocAgent + 实现 Agent |
| `代码参考.md`   | 代码模式：API 签名、用法范例、注意事项                   | DesignDocAgent + 实现 Agent |
| `接口契约.md` | 接口契约总账：接口 ID、method、path、请求/响应、错误码、权限、状态 | 后端/Mock/联调方 + 实现 Agent |
| `前端设计.md`  | 模块实现方案：引用代码参考，给出具体代码          | 实现 Agent                  |
| `编码任务.md` | 编码任务包：代码级完成标准、门禁命令、回归检查、停止条件 | 实现 Agent                  |

### 2.2.1 中文产物命名

新生成的业务文件和业务文件夹一律使用中文名，Manifest 只记录中文路径。不要主动查找、生成、迁移或写回旧英文产物名；如果用户显式提供旧英文文件路径，可作为普通输入读取，但不得把旧英文名作为默认规则。

稳定 ID 不中文化：`M{N}`、`API-Mx-xxx`、`T-Mx-xx` 必须保持机器可读。

### 2.3 默认不生成的治理产物

| 产物               | 默认不生成原因                                     | 例外条件                        |
| ------------------ | -------------------------------------------------- | ------------------------------- |
| `spec.md`          | 与 需求说明.md 重叠，合并进 需求说明       | 无                              |
| `facts.md`         | 易与 需求说明/代码参考 形成重复事实源    | 严格审计或证据需单独归档        |
| `tasks.md`         | 与 编码任务.md 重叠                        | 默认不恢复                      |
| `ci-gate.md`       | 门禁命令先进入 编码任务.md，避免早期空壳化 | 进入真实实现、测试、CI 修复阶段 |
| `agent-prompts.md` | 单 Agent 执行时会增加阅读成本                      | 多 Agent 并行协作               |
| `fact-set.yaml`    | 人读/AI 读场景下维护成本高，易漂移                 | 脚本、检查器或多 Agent 自动消费 |

### 2.4 主流程（v2.3 模式交互版）

```text
initial → mode_select
        ├─ module-split
        │   ├─ 输入：PRD / MasterGo / 原型 / 截图
        │   ├─ 输出：Manifest 模块索引.md + M{N}-{中文模块名} 模块文件夹
        │   └─ 停止：等待人工审查、补图和冲突确认
        ├─ module-design
        │   ├─ 输入：模块索引.md#M{N} + 当前模块 需求/界面/原型/审查记录 + 代码库
        │   ├─ clarify：只澄清当前模块的边界、异常、权限、交互
        │   ├─ evidence / code_ref：Codegraph 提取当前模块需要的代码模式
        │   ├─ api_contract：默认生成/更新根级 接口契约.md
        │   ├─ design / analyze：生成并自审 前端设计.md
        │   └─ task / verify：生成并验证 编码任务.md
        └─ implement
            ├─ 输入：编码任务.md + 必要前端设计/代码参考章节
            └─ 输出：代码改动、验证结果、Regression Check、Defensive Code
```

## 3. Agent 设计

### 3.1 InputCollectorAgent

| 项目 | 内容                               |
| ---- | ---------------------------------- |
| 目标 | 收集并归档所有输入材料             |
| 输入 | 用户链接、本地文件、截图、补充说明 |
| 输出 | 输入清单、访问状态、材料缺口       |
| 禁止 | 不做需求推断，不生成设计结论       |

### 3.2 RequirementStructAgent

| 项目 | 内容                                   |
| ---- | -------------------------------------- |
| 目标 | 把 PRD 转成结构化 需求说明         |
| 输入 | PRD、已有 Spec（如有）、业务补充       |
| 输出 | `需求说明.md`（含原 spec.md 内容） |
| 禁止 | 不写实现细节，不选择组件库             |

### 3.3 CodeReferenceAgent（v2 核心新增）

| 项目         | 内容                                            |
| ------------ | ----------------------------------------------- |
| 目标         | 从现有代码中提取项目级代码模式                  |
| 输入         | 代码库路径、组件库路径                          |
| 输出         | `代码参考.md`                             |
| 首选 MCP     | Codegraph MCP                                   |
| **关键约束** | **此 Agent 的输出是 DesignDocAgent 的前置依赖** |

提取模式（按项目技术栈调整）：

| 序号 | 模式       | Codegraph 入口查询                              |
| ---- | ---------- | ----------------------------------------------- |
| 1    | 列表页工厂 | `codegraph_context("最完整的列表页")`           |
| 2    | 表单项工厂 | `codegraph_context("表单字段工厂 useFormItem")` |
| 3    | 工具栏插件 | `codegraph_search("usePlugin")`                 |
| 4    | 抽屉/弹窗  | `codegraph_context("新增编辑抽屉")`             |
| 5    | 脏数据检测 | `codegraph_context("脏数据 diff changed")`      |
| 6    | 请求层     | `codegraph_context("请求 request APIMap")`      |
| 7    | 路由注册   | `codegraph_context("路由 route config")`        |
| 8    | 权限控制   | `codegraph_context("权限 permission codes")`    |
| 9    | 状态管理   | `codegraph_context("状态管理 store")`           |
| 10   | 导出功能   | `codegraph_context("导出 export download")`     |
| 11   | 审批流     | `codegraph_context("审批 approval")`            |

**每个模式必须输出**：
- 完整的 TypeScript 签名（从代码中提取，不是猜测）
- 至少 1 个来自现有代码的真实使用范例
- 注意事项（踩坑点、特殊处理、与其他模式的交互）

### 3.4 PrototypeFactAgent

| 项目     | 内容                                         |
| -------- | -------------------------------------------- |
| 目标     | 从 MasterGo/原型中提取页面事实               |
| 输入     | MasterGo 图层链接、原型链接、截图            |
| 输出     | 布局数据（注入 前端设计.md 对应模块） |
| 首选 MCP | MasterGo MCP、Chrome DevTools MCP            |

v2 DSL 处理规则：
- DSL ≤ 50KB：直接内联
- DSL > 50KB：提取关键结构（组件树、尺寸、布局），忽略纯装饰节点

### 3.5 ApiContractAgent

| 项目         | 内容                                                   |
| ------------ | ------------------------------------------------------ |
| 目标         | 默认维护根级接口契约总账                              |
| 输入         | `需求说明.md`、接口线索、澄清记录、API 文档（如有） |
| 输出         | `接口契约.md`                                      |
| **关键约束** | **接口契约事实只写在 接口契约.md，模块设计只引用接口 ID** |

接口状态只能使用 `Draft`、`Reviewed`、`Confirmed`、`Changed`。真实联调依赖 `Confirmed`；本地骨架或 Mock 至少依赖 `Reviewed`。无新增/变更接口的模块，必须在 `接口契约.md` 的“模块接口覆盖记录”中记录“无新增/变更接口”，不得写入接口总览。

### 3.6 DesignDocAgent

| 项目         | 内容                                             |
| ------------ | ------------------------------------------------ |
| 目标         | 生成模块实现方案                                 |
| 输入         | `需求说明.md`、`代码参考.md`、DSL 数据、`接口契约.md` |
| 输出         | `前端设计.md`                             |
| **关键约束** | **必须引用 代码参考.md 的章节号；涉及接口时只引用接口 ID** |

v2 模块结构：
- P0 核心模块：**完整版**（所有代码骨架、文件清单、路由/API/Store 注册代码）
- 其余模块：**差异版**（只写与完整版的差异部分）

### 3.7 TaskAgent

| 项目         | 内容                                      |
| ------------ | ----------------------------------------- |
| 目标         | 生成代码级完成标准的任务包                |
| 输入         | `前端设计.md`、`代码参考.md`、`接口契约.md` |
| 输出         | `编码任务.md`                     |
| **关键约束** | **完成标准必须是代码级断言；涉及接口时必须引用接口 ID 和状态停止条件** |

完成标准示例：
- ❌ `列表加载+筛选+分页`
- ✅ `createPage 返回 { api: { list, autoLoad:true }, form: { body: { fieldA(text), fieldB(vendor:multi) } }, table: { plugins: [usePluginBtn, usePluginDel], vxeProps: { height:'100%' } } }`

## 4. MCP 设计

### 4.1 MCP 优先级

| 场景                   | 首选 MCP            | 降级方式      |
| ---------------------- | ------------------- | ------------- |
| 代码结构、符号、调用链 | Codegraph MCP       | 只读 RTK 命令 |
| PRD/原型/组件库文档    | Chrome DevTools MCP | 用户导出文件  |
| MasterGo 图层          | MasterGo MCP        | 用户导出 DSL  |

### 4.2 CodeReferenceAgent 的 Codegraph 使用

```text
1. codegraph_status → 确认索引就绪
2. codegraph_files → 了解项目结构
3. codegraph_context("关键模式") → 找到模式入口
4. codegraph_explore(入口符号) → 深入读取实现
5. codegraph_node(具体符号) → 获取签名和源码
```

## 5. Prompt 设计

### 5.1 ClarifyAgent 角色约束（v2.3：仅 module-design 强制）

```text
你是需求澄清专家。
你的任务不是写文档，而是通过苏格拉底式提问消除需求歧义。
每轮最多 3-5 个问题。分段呈现，等用户回答后再继续。
问题聚焦：边界条件、异常路径、权限差异、模块交互。
module-design 禁止跳过澄清直接生成文档。
module-split 不强制澄清，只记录待确认问题到 审查记录.md。
```

### 5.2 CodeReferenceAgent 角色约束

```text
你是代码模式提取专家。
你的任务是从现有项目中提取可复用的代码模式，而不是设计新模式。
每个模式必须有：完整签名、真实代码范例、注意事项。
禁止编造签名。禁止使用伪代码。所有代码必须来自实际文件。
```

### 5.3 DesignDocAgent 角色约束

```text
你是前端设计专家。
你必须先读取 代码参考.md，理解项目怎么写的，然后设计新模块怎么写。
每个模块必须引用 `代码参考.md` 的章节号。
禁止使用"按 PRD""参考设计稿"等不可执行描述。
分段呈现：每段 1 个模块，等用户确认后再继续。
```

### 5.4 AnalyzeAgent 角色约束（v2.3）

```text
你是设计书审查专家。
你的任务是用 6 项检查验证 design 能否让 AI 直接写代码。
最关键的一项：只看 代码参考.md + 接口契约.md + 前端设计.md，你能否写出该模块完整的 index.tsx？
如果不能，标注缺失信息并要求回补。
自审结果必须用表格输出，不可用模糊的"基本满足"。
```

### 5.5 TaskAgent 角色约束

```text
你是任务分解专家。
每个任务必须满足：2-5 分钟粒度、≤ 3 个文件、≤ 100 行新代码。
完成标准必须能转成 assert 伪代码，不可用模糊描述。
每个任务必须包含验证步骤。
```

## 6. 质量门禁（v2.3）

### 6.0 module-split 门禁

| 检查项   | 标准                                                                          |
| -------- | ----------------------------------------------------------------------------- |
| Manifest | `模块索引.md` 只维护模块身份、路径、状态和索引，不复制详细事实            |
| 模块索引 | `模块索引.md` 包含模块清单、依赖、跨模块规则、补图记录和待确认问题        |
| 模块文件 | 每个模块目录包含 `需求.md`、`界面.md`、`原型.md`、`审查记录.md` |
| 事实边界 | 模块文件只记录事实和待确认项，不包含代码设计方案                              |
| 冲突记录 | PRD/UI/原型冲突进入 `审查记录.md`                                         |
| Evidence | 所有事实带 Evidence                                                           |

### 6.1 需求说明.md 门禁

| 检查项   | 标准                                            |
| -------- | ----------------------------------------------- |
| 模块来源 | 以 `模块索引.md` 和当前模块文件夹为主要输入 |
| 澄清记录 | 当前模块关键澄清问题已回答或已标 TODO           |
| 模块覆盖 | 当前模块范围、依赖、字段、权限已覆盖            |
| 验收标准 | 当前模块有 Given/When/Then                      |

### 6.2 代码参考.md 门禁

| 检查项      | 标准                                |
| ----------- | ----------------------------------- |
| 模式覆盖    | ≥ 10 个核心代码模式                 |
| 签名完整性  | 每个模式有完整 TypeScript 签名      |
| 范例真实性  | 每个模式有 ≥ 1 个真实代码范例       |
| import 路径 | 所有路径 Verified（来自 Codegraph） |

### 6.2.1 接口契约.md 门禁

| 检查项       | 标准                                                        |
| ------------ | ----------------------------------------------------------- |
| 路径位置     | 位于设计根目录，模块设计目录默认用 `../../接口契约.md` 引用 |
| 接口总览     | 只包含真实接口 ID，不把“无新增/变更接口”当作接口行          |
| 覆盖记录     | 无新增/变更接口的模块写入“模块接口覆盖记录”                 |
| 状态语义     | Draft / Reviewed / Confirmed / Changed 使用正确             |
| Draft path   | Draft 接口可用 Proposed path，但必须标 Inferred / 待确认    |
| 并行更新     | 只更新当前模块 section 和全局总览对应行                     |

### 6.3 前端设计.md 门禁

| 检查项              | 标准                                 |
| ------------------- | ------------------------------------ |
| 代码参考 引用 | 每个模块引用 ≥ 1 个章节号            |
| 文件清单            | 每个模块给出可创建的文件列表         |
| 代码骨架            | P0 模块有完整可粘贴代码              |
| DSL 数据            | 有 MasterGo 链接的模块注入了布局数据 |

### 6.4 /analyze 自审门禁（v2.3）

| 检查项   | 标准                                          | 不通过动作   |
| -------- | --------------------------------------------- | ------------ |
| 模拟实现 | 只看 代码参考 + 接口契约 + 前端设计 能写出 index.tsx | 回补代码骨架 |
| 代码粘贴 | "可粘贴"代码语法正确、import 路径存在         | 修正代码     |
| 完成标准 | 每个任务完成标准能写成 assert                 | 改写为断言   |
| 交叉引用 | 前端设计引用的章节号真实存在                   | 修正引用     |
| 缺失模式 | 无模块需要未覆盖的代码模式                    | 回补章节     |
| 粒度检查 | 每个任务 2-5 分钟                             | 拆分任务     |

**退出条件**：≥ 4/6 通过 → 继续；≤ 3/6 → 回退 evidence 重新采集

### 6.5 编码任务.md 门禁

| 检查项   | 标准                                 |
| -------- | ------------------------------------ |
| 完成标准 | 每个任务是代码级断言                 |
| API 引用 | 涉及接口的任务引用 `接口契约.md` 中真实接口 ID；无接口任务写“无” |
| 依赖关系 | 任务间依赖正确标注                   |
| 任务粒度 | 每个任务 ≤ 3 文件、≤ 100 行          |
| 验证步骤 | 每个任务有明确验证步骤               |

### 6.6 /verify 最终验证门禁（v2.3）

| 检查项   | 标准                 | 不通过动作     |
| -------- | -------------------- | -------------- |
| 粒度检查 | 2-5 分钟/任务        | 拆分过大的任务 |
| 精确路径 | 无 TODO 路径         | 补全路径       |
| 验证步骤 | 每任务有验证步骤     | 补全步骤       |
| 可断言   | 每标准可转 assert    | 改写标准       |
| 依赖完整 | 前置依赖已定义       | 修正依赖       |
| 回归覆盖 | 回归清单覆盖已有功能 | 补全清单       |
| 接口引用 | 涉及接口的任务引用真实接口 ID；无接口任务写“无” | 修正 API 引用 |
| 接口状态 | Draft 不进真实联调；Mock/本地骨架至少 Reviewed；真实联调必须 Confirmed | 修正停止条件 |
| 契约路径 | 模块设计目录默认可解析 `../../接口契约.md` | 修正路径 |
| Manifest | `模块索引.md#M{N}` 中模块锚点、模块目录、输入文件、设计输出目录和根级 API 契约路径可解析 | 修正 Manifest |

**退出条件**：6/6 通过 → done；有不通过 → 回退 task 修正

## 7. 风险与优化

| 风险                      | v2.3 处理方式                                                                 |
| ------------------------- | ----------------------------------------------------------------------------- |
| 模式判断不清              | 先询问用户选择 `module-split` / `module-design` / `implement`                 |
| Codegraph 未初始化        | 提示初始化；降级 Evidence 标记 Fallback                                       |
| MasterGo DSL 数据过大     | 提取关键结构，生成精简版                                                      |
| 项目代码模式不统一        | 选最具代表性的实现，标注差异                                                  |
| 新需求无现有参考          | 标注为"新模式"，给出推荐实现                                                  |
| design 过度自然语言化     | 强制代码骨架、字段字典表、API 表                                              |
| **module-split 过早设计** | 停止生成设计书，只输出模块事实切片和 review-notes                             |
| **用户拒绝澄清**          | 在 module-design 记录"用户跳过澄清"到 TODO，基于最佳实践推断，标注 Unverified |
| **自审循环**              | 连续 2 轮自审 ≤ 3/6 → 标记阻塞，要求用户介入                                  |
| **任务拆分过细**          | 如果某模块拆出 > 20 个任务 → 合并同类任务，保持 ≤ 15                          |
| **分段呈现拖慢节奏**      | 用户明确说"一次性输出"时跳过分段                                              |
