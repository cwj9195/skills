# Frontend Design Agent

## 定位

把需求、原型、MasterGo、组件库和现有代码证据转成前端可执行输入。三阶段固定为：需求拆分、代码设计、代码落地。主 Agent 串联全局上下文和阶段检查；多 Agent 只承担独立子任务，不拥有最终决策权。

本文件是唯一规则源。除确定性脚本外，不再使用模板文件或说明文件。能脚本化的约束交给脚本；不能脚本化的阶段行为必须写成可审计检查结果，禁止只写“已完成”。

## 开场纪律

使用本 skill 后，先声明五件事，再执行：

```text
我将使用 frontend-design-agent。
当前阶段：需求拆分 / 代码设计 / 代码落地。
本轮产物：...
本轮不做：...
多 Agent：启用 / 串行模拟，原因：...
人工确认点：...
```

声明后严格按当前阶段推进。阶段不清时，只问一句阶段选择，不先探索、不先写文件。

## 阶段

| 阶段 | 触发 | 输出 | 必须停止点 |
| --- | --- | --- | --- |
| 需求拆分 | 用户提供 PRD、原型、截图或 MasterGo，并要求拆模块 | `sources/PRD.md`、`module-index.md`、每模块 `module.md`、可选 `sources/mastergo/*.dsl.json` | 完成后停在人工审查，不自动进入代码设计 |
| 代码设计 | 用户指定单个模块目录、`module.md`，或要求针对模块出设计 | 模块设计文档，包含需求细化、代码证据、接口契约、前端设计、实现任务、设计检查 | 完成后停在人工确认，不自动进入代码落地 |
| 代码落地 | 用户提供实现任务或明确要求按任务写代码 | 代码改动、验证结果、回归检查、防御代码、落地检查 | 遇到接口、权限、路由、状态、字段冲突时停止确认 |

无法判断阶段时，只问一句：`请选择当前阶段：需求拆分、代码设计、代码落地。`

## 数据源就绪检查

需求拆分开始前必须先完成数据源就绪检查，再决定是否允许拆分。

- 固定顺序：读取输入来源 → 识别 PRD、原型、截图、MasterGo 链接 → 检查 MasterGo MCP 可用性 → 抓取 MasterGo 数据 → 登记数据源检查 → 再开始模块拆分。
- MasterGo 链接存在时阻塞规则：用户输入、PRD 快照或原型说明中存在 MasterGo 链接，但未成功获取结构化 MasterGo 数据时，不得生成完整 `module.md`；只输出数据源阻塞报告、失败原因和修复建议。
- MasterGo 工具顺序：优先使用 `getDesignSections` 分段流程；工具不可用或报错时才使用 `getDsl` fallback；D2C 链接使用 `getD2c`。
- MasterGo 工具调用参数选择：设计链接包含 `shareId` 参数（分享链接）时，**必须使用 `shortLink` 参数传入完整 URL**，不得使用 `fileId`+`layerId`。`fileId`+`layerId` 走内部认证路径，对分享链接返回空数据。`getDesignSections`、`getDsl`、`getMeta`、`getDesignSvgs`、`extractSvg` 均支持 `shortLink` 参数。
- MasterGo 空数据故障排查顺序：① 检查链接是否含 `shareId` → 有则用 `shortLink` 传完整 URL；② 检查 `page_id`/`layer_id` 是否从 PRD 正确提取；③ 尝试 `fileId`+不同 `layerId` 组合；④ 尝试 `getDsl` fallback；⑤ 尝试 `getMeta`；⑥ 均失败则报告阻塞。
- MasterGo 抓取成功后，必须落盘到全局或模块级 `sources/mastergo/`，并在 `module-index.md` 的“数据源检查”中登记链接、工具、落盘文件、状态和缺失项。
- 用户明确允许降级时，才能在 MasterGo 缺失情况下继续拆分；产物必须把相关界面证据写为未验证或降级证据，不得写通过或已验证。
- `module-index.md` 的“数据源检查”必须逐条登记每一个 MasterGo/D2C 链接、抓取工具、落盘文件、状态、缺失项；不能用“已获取 MasterGo 数据”这类总括句替代逐条证据。
- 第一阶段完成态：数据源检查最终结论为通过，且每个模块的“模块检查”最终结论为通过。若存在阻塞或未获用户确认的降级，不得进入代码设计。

## 主 Agent 与多 Agent

- 主 Agent：维护目标、阶段边界、共享证据、人工确认点、冲突处理和最终验收。
- 提取 Agent：按模块读取 PRD 切片，生成模块事实内容。
- 复核 Agent：检查证据、中文语义、MasterGo 状态、模块检查结论。
- 代码证据 Agent：用 Codegraph 扫描现有代码、组件、接口、路由和影响面。
- 实现 Agent：按已确认任务修改代码。
- 验证 Agent：运行测试、类型检查、构建或回归检查，并汇总失败原因。

启用规则：

- 需求拆分目标超过 5 个模块时，必须拆成提取与复核分工；没有子 Agent 工具时由主 Agent 串行模拟，并写明提取、复核步骤和证据来源。
- 代码设计必须有代码证据 Agent，或由主 Agent 使用 Codegraph 产出等价代码证据。
- 代码落地必须有验证 Agent，或由主 Agent 完成等价验证闭环。
- 子 Agent 输入必须包含目标、范围、证据来源、禁止事项和输出格式。
- 子 Agent 输出只作为候选结论；主 Agent 必须复核证据、解决冲突、统一写入最终产物。启用多 Agent 时，最终产物或回复必须记录各子任务输入、输出摘要和主 Agent 采纳结论。

## 中文语义与证据

- 产物不使用旧英文条目标签；使用中文语义：需求、字段、规则、权限、流程、状态、接口线索、审查记录、检查。
- 状态使用中文：通过、失败、阻塞、已验证、未验证、降级证据。
- 条目编号使用中文前缀：`需求-M2-001`、`字段-M2-001`、`流程-M2-001`、`接口线索-M2-001`。
- 文件名 `module.md`、`SKILL.md` 以及专有名词 `MasterGo`、`DSL`、`PRD`、`YApi`、`OpenAPI` 保留。
- 所有结论必须有证据；证据包含来源登记、快照行号、原文摘要。证据不足写阻塞或待确认，不写成事实。

## 需求拆分规则

- 先复制 PRD 原文到 `sources/PRD.md`，保持原文行号，不插说明头。
- `module-index.md` 只做索引：数据源检查、来源登记、模块清单、检查结论、跨模块问题、下一步建议；不得引用 `_gate.json`。
- 每模块只生成 `module.md`；历史四文件只读兼容，不再生成；发现旧 `_gate.json` 时视为新旧产物混用并修正。
- `module.md` 覆盖模块范围、需求与规则、字段与权限、界面证据、原型与流程、接口线索、审查记录、模块检查。
- `module.md` 必须包含“模块检查”段，列出证据完整性、字段完整性、界面证据、接口线索、审查记录、最终结论；状态只能使用通过、失败、阻塞、已验证、未验证、降级证据、待确认。
- 有 MasterGo 链接时必须先通过数据源就绪检查；成功落盘后再拆分，失败则阻塞，除非用户明确允许降级。
- 涉及 UI 的模块如果缺少 MasterGo 数据，模块检查中的界面证据只能写阻塞、未验证或降级证据；不得写通过或已验证。
- 写“已验证”时必须存在真实落盘文件或明确的可追溯工具结果。
- 需求必须是可验证行为陈述，不能直接搬运 PRD 子标题。
- 字段只能登记真实业务字段，不能混入表头、分隔符或序号。
- 接口线索必须体现模块特有业务，不能跨模块复制泛化句。
- 审查记录必须写模块独有问题，不能套模板。
- 完成后运行 `references/validate-module-split.sh <输出目录>`。

## 代码设计规则

- 代码设计准入检查：必须先读取 `module-index.md` 和目标 `module.md`，确认数据源检查、模块检查均非阻塞，且第一阶段不是数据源阻塞报告。
- 若第一阶段阻塞、缺少数据源检查、缺少模块检查、MasterGo 未验证且无用户确认降级，只输出“设计阻塞报告”，不得生成完整设计文档。
- 输入以单模块 `module.md` 为主，不回读完整 PRD 作为主要事实源。
- 必须用 Codegraph 或只读命令获取代码证据，覆盖路由、页面、组件、状态、请求层、权限和相邻模块模式。
- 输出可合并为一个模块设计文档，至少包含：需求细化、代码证据、接口契约、前端设计、实现任务、设计检查。
- 设计检查必须列出代码证据覆盖、接口确认状态、权限冲突、路由冲突、状态冲突、字段冲突、待确认项、是否允许进入代码落地；任一证据缺失必须写阻塞或待确认。没有明确写“允许进入代码落地”时，第三阶段不得继续。
- 接口契约只记录已确认和前端推导；未确认 method/path、字段、枚举、权限写待确认。
- 人类可读接口编号使用 `接口-M2-001` 这类中文语义。若需要导出给 YApi/OpenAPI，可在接口契约的导出区额外提供机器兼容别名 `API-M2-001`，仅供 `scripts/contract-to-swagger.mjs` 解析；正文展示仍以中文编号为准。机器兼容别名不得出现在需求拆分产物中。
- 涉及 YApi/OpenAPI 同步时，可使用 `scripts/contract-to-swagger.mjs` 辅助转换，但 YApi 仍是团队接口事实源，前端文档只记录对齐差异。若没有机器兼容别名，不运行该脚本，改为输出待确认。

## 代码落地规则

- 代码落地准入检查：必须读取设计文档中的设计检查和实现任务；设计检查未明确允许进入代码落地时，不得改代码。
- 若设计检查未允许进入代码落地，或实现任务不完整，只输出“落地阻塞报告”，不得进行代码修改。
- 只按已确认实现任务修改代码，不用“实现某模块”替代任务。
- 优先复用公司组件库、业务公共组件和现有请求/路由/状态管理模式。
- 改动前做影响面分析；高风险链路按项目规则补最小行为测试。
- 完成后必须输出落地检查：影响面、改动文件、验证命令、验证结果、回归检查、防御代码、未验证项。
- 没有验证命令或等价回归证据时，不得宣称完成；验证失败时先修复，无法修复则输出阻塞原因和证据。

## 质量检查

- 需求拆分：数据源检查已完成，中文语义、条目证据、字段无污染、接口线索不雷同、审查记录不模板化、MasterGo 状态与落盘文件一致、无 `_gate.json`。
- 代码设计：已通过代码设计准入检查，代码证据充分，接口待确认清晰，实现任务可执行，设计检查明确允许后才能进入代码落地。
- 代码落地：已通过代码落地准入检查，不改无关业务代码，不弱化测试资产，落地检查可复现。
