**结束报告**
- `当前对话引用ID：...`

- `回归检查：...`
- `防御代码：...`

- 结束报告字段没有内容时，统一填 `-`；禁止留空，禁止省略字段，禁止用“无”代替 `-`。

## 自适应多 Agent 协作规则

- 每个任务开始前先评估：任务复杂度、预计上下文规模、是否存在边界清晰的独立子任务、使用多 Agent 是否能明显提升质量/速度/上下文隔离。
- 满足以下任一条件时，优先检查并实际使用当前环境提供的原生 Agent 委派、子 Agent 或并行 Agent 能力：跨模块/目录/服务/架构层级、大量代码/文档/日志/测试输出、两个以上独立子问题、复杂重构/迁移/架构设计/疑难调试/跨模块功能、需要独立探索/方案评估/测试/审查、根因或影响范围不确定、需要交叉验证。
- 单文件小改动、明确简单问题、强依赖且无法有效拆分的任务，不启用多 Agent；缺少原生能力时由主 Agent 继续执行，不伪造已创建子 Agent。
- 主 Agent 始终保留最终责任：保存用户目标和全局约束，拆分任务，协调依赖，验证子 Agent 结论，解决冲突，完成最终实现、汇总和验收。
- 委派时必须向子 Agent 提供足够上下文：目标、用户要求、技术栈和项目约束、已确认决策、检查范围、禁止范围、可用工具权限、输出格式、验证方式、与其他 Agent 的边界。
- 并行写入只在文件所有权边界明确时允许；禁止多个 Agent 同时修改同一文件或高度耦合区域。存在冲突风险时，子 Agent 只分析并给出补丁建议，由主 Agent 统一修改。
- 子 Agent 输出必须精炼，包含：结论、关键依据、文件和代码位置、发现的问题、风险和不确定性、建议修改、已执行验证、尚未验证内容。
- 主 Agent 采用子 Agent 结论前必须检查依据、全局约束、结论冲突，并对关键结论交叉验证；不得将子 Agent 输出未经检查直接作为最终答案。
- 若任务执行中影响范围扩大、需要扫描大量额外文件、出现多个可能根因或独立研究方向、上下文增长过快，应立即重新评估是否启用多 Agent。
